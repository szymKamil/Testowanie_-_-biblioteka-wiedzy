# Wprowadzenie do historii i działania Selenium

Selenium to pakiet *open source* składający się z zestawu bibliotek i narzędzi, które umożliwiają automatyzację przeglądarek internetowych. Możemy postrzegać Selenium jako projekt parasolowy z trzema podstawowymi komponentami: **WebDriver**, **Grid** i **IDE** (zintegrowane środowisko programistyczne), w którym tworzymy testy/customowe biblioteki testowe. 

`Selenium WebDriver` to biblioteka, która umożliwia programowe sterowanie przeglądarkami. W ten sposób możemy używać `Selenium WebDriver` do poruszania się po stronach internetowych i interakcji ze stronami internetowymi (np. klikanie linków, wypełnianie formularzy itp.), tak jak robiłby to prawdziwy użytkownik, w sposób zautomatyzowany. Podstawowym zastosowaniem `Selenium WebDriver` jest zautomatyzowane testowanie aplikacji internetowych. Inne zastosowania Selenium obejmują automatyzację zadań administracyjnych opartych na sieci lub *skrobanie* stron internetowych.

**Historia powstania Selenium w skrócie:**

- **2004**: Jason Huggins tworzy "JavaScriptTestRunner" (późniejsze Selenium Core) w ThoughtWorks do automatyzacji testów aplikacji webowych. Narzędzie zostaje udostępnione jako open source.
- **2005**: Dan Fabulich i Nelson Sproul znacząco przyczyniają się do rozwoju Selenium Remote Control (RC), umożliwiając sterowanie testami z różnych języków programowania.
- **2007**: Simon Stewart w ThoughtWorks rozpoczyna pracę nad WebDriver, alternatywnym podejściem do automatyzacji przeglądarek, komunikującym się bezpośrednio z ich API.
- **2008**: Philippe Hanrigou tworzy Selenium Grid, pozwalające na równoległe uruchamianie testów na wielu maszynach.
- **2009**: Na konferencji Google Test Automation Conference zapada decyzja o połączeniu projektów Selenium i WebDriver, co daje początek Selenium 2.0 (Selenium WebDriver).

Selenium WebDriver to biblioteka, która umożliwia automatyczne sterowanie przeglądarkami internetowymi. W tym celu zapewnia wieloplatformowe API w różnych rozwiązaniach językowych. Sterowniki (np. chromedriver, geckodriver itp.) to zależne od platformy pliki binarne, które odbierają polecenia ze skryptu `WebDriver` i tłumaczą je na język specyficzny dla przeglądarki. W pierwszych wersjach Selenium WebDriver (tj. w Selenium 2) polecenia te (znane również jako protokoły Selenium) były komunikatami JSON przesyłanymi przez HTTP (tzw. JSON Wire Protocol). Obecnie ta komunikacja (nadal JSON przez HTTP) jest zgodna ze standardową specyfikacją o nazwie W3C WebDriver https://www.w3.org/TR/webdriver2/. Specyfikacja ta jest preferowanym protokołem Selenium od wersji Selenium 4.

![obraz.png](Selenium%20JS/obraz.png)

Podstawowa architektura Selenium WebDriver składa się z trzech poziomów. Skrypt wykorzystujący `API Selenium WebDriver` (Java, JavaScript, Python, Ruby lub C#). Ten skrypt wysyła polecenia W3C WebDriver do drugiej warstwy, w której znajdujemy sterowniki (`driver`). Powyższy rysunek pokazuje konkretny przypadek użycia chromedriver (do sterowania Chrome) i geckodriver (do sterowania Firefox). Trzecia warstwa zawiera przeglądarki internetowe. W przypadku Chrome natywna przeglądarka jest zgodna z protokołem DevTools. DevTools to zestaw narzędzi deweloperskich dla przeglądarek opartych na silniku renderującym Blink, takich jak Chrome, Chromium, Edge czy Opera. Protokół DevTools opiera się na komunikatach JSON-RPC i umożliwia inspekcję, debugowanie i profilowanie tych przeglądarek. W przeglądarce Firefox natywna obsługa automatyzacji wykorzystuje protokół Marionette. Marionette to zdalny protokół oparty na JSON, umożliwiający instrumentowanie i kontrolowanie przeglądarek internetowych opartych na silniku Gecko (takich jak Firefox).

**Selenium Grid**

Drugim projektem z rodziny Selenium jest `Selenium Grid` powstały w 2008 roku. `Selenium Grid` to grupa hostów sieciowych, które zapewniają infrastrukturę przeglądarek dla `Selenium WebDriver`. Infrastruktura ta umożliwia (równoległe) wykonywanie skryptów (inaczej - testów) `Selenium WebDriver` na zdalnych przeglądarkach o różnym charakterze (typach i wersjach) w wielu systemach operacyjnych.

![obraz.png](Selenium%20JS/obraz%201.png)

Architektura `Selenium Grid` składa się z grupy węzłów (*node*) zapewniających przeglądarki używane przez skrypty Selenium. Węzły te mogą korzystać z różnych systemów operacyjnych z różnymi zainstalowanymi przeglądarkami. Centralnym punktem wejścia do tej siatki jest Hub (znany również jako Selenium Server). Ten komponent po stronie serwera śledzi węzły (ich stan) i proxy żądań ze skryptów Selenium. Podobnie jak w `Selenium WebDriver`, specyfikacja W3C WebDriver jest standardowym protokołem komunikacji między tymi skryptami a Hubem.

Architektura hub-nodes w Grid jest dostępna od czasu Selenium 2. Jest ona również obecna w Selenium 3 i 4. Niemniej jednak może prowadzić do wąskich gardeł wydajności, jeśli liczba żądań do huba jest wysoka. Selenium 4 zapewnia w pełni rozproszoną wersję Selenium Grid, aby uniknąć tego problemu. Architektura ta implementuje zaawansowane mechanizmy równoważenia obciążenia, aby uniknąć przeciążenia jakiegokolwiek komponentu.

Selenium IDE

`Selenium IDE` jest ostatnim podstawowym komponentem pakietu Selenium powstałym w 2006 roku. `Selenium IDE` jest narzędziem, które implementuje tzw. technikę automatyzacji Record and Playback (R&P). Przy jego wykorzystaniu test może zostać wytworzony w sposób manualny (wyklikany) i wyeksportowany do kodu (np. Java) wykorzystującego bibliotekę `WebDriver`. 

Projekt ten funkcjonujący jako wtycznaa nie został dostosowany do nowych wymogów sklepu Chrome, działa tylko w [Firefox](https://www.selenium.dev/selenium-ide/). Istnieje także działająca wersja niezależna od przeglądarek - w formie narzędzia uruchomieniowego. Projekt ten nie jest jednak rozwijany.

# Narzędzia kompilacji

Ważnym komponentem jest narzędzie do kompilacji. Narzędzia kompilacji to programy narzędziowe do automatyzacji tworzenia aplikacji wykonywalnych z kodu źródłowego. Narzędzia te ułatwiają zarządzanie projektem w zakresie zarządzania zależnościami, kompilacji, pakowania, wykonywania testów i wdrażania. Ogólnie rzecz biorąc, narzędzia do kompilacji są wygodnym sposobem zautomatyzować rozwój projektów oprogramowania, zarówno na serwerach kompilacji (np. GitHub Actions), jak i na maszynach deweloperskich. Dlatego też niezbędne jest korzystanie z narzędzia do kompilacji do skonfigurowania projektu. Istnieją dwie główne alternatywy wykorzystywane w Selenium: 

- `Maven` - otwarte narzędzie do automatyzacji kompilacji utrzymywane przez Apache Software Foundation. Jest używane głównie w projektach Java, chociaż obsługuje również inne języki, takie jak takie jak C#, Ruby czy Scala.
- `Gradle` - narzędzie open source do automatyzacji kompilacji oprogramowania. Obsługuje obsługuje Javę i inne języki, takie jak Kotlin, Groovy, Scala, C/C++ lub JavaScript.

## Project Layout

Układ projektu to struktura katalogów używana do przechowywania różnych zasobów projektu oprogramowania (np. kodu źródłowego, plików binarnych, zasobów statycznych itp.) Maven i Gradle używają równoważnego układu dla projektów Java.

![image.png](Selenium%20JS/image.png)

Reszta układu projektu różni się w obu narzędziach kompilacji. Pierwszą różnicą jest plik konfiguracyjny. Z jednej strony plik ten jest unikalny i nazywa się pom.xml (Project Object Model) w Maven. W Gradle istnieją natomiast dwa pliki konfiguracyjne, zwane `settings.gradle` i `build.gradle`. Drugą różnicą między Maven i Gradle jest folder wyjściowy. W obu przypadkach narzędzia kompilacji tworzą ten folder, aby zapisać wynikową kompilację (tj. skompilowane klasy, wynikowe spakowane pliki itp.). Nazwa tego folderu to *target* w Maven i **build** w Gradle. Wreszcie, Gradle zawiera zestaw folderów i plików dla tak zwanego wrappera Gradle. Ten wrapper to plik skryptu (o nazwie gradlew dla systemów uniksowych i gradlew.bat dla Windows), który zapewnia następujące korzyści: 

- Budowanie projektu bez instalowania Gradle na lokalnej maszynie.

- Wymaganie użycia danej wersji (która może być inna niż lokalnie zainstalowana instancja Gradle).

- Łatwa aktualizacja do nowej wersji poprzez zmianę artefaktów wrappera (w folderze gradle/wrapper).

Więcej informacji na temat kompilatorów można znaleźć w ich oficjalnej dokumentacji [Maven](https://maven.apache.org/guides/) i [Gradle](https://docs.gradle.org/current/userguide/userguide.html).

**Zależności** 

Zależności projektu oprogramowania to wymagane biblioteki lub wtyczki. Narzędzia do kompilacji umożliwiają między innymi zautomatyzowane zarządzanie zależnościami projektu. W tym celu musimy określić współrzędne takich zależności w pliku konfiguracyjnym projektu. Współrzędne projektu Java to grupa trzech etykiet, które jednoznacznie identyfikują ten projekt (np. biblioteka, wtyczka itp.), W przypadku Maven zależności definiujemy w pliku `pom.xml`. Przydatne biblioteki opisane są na końcu dokumentu. 

# Testy automatyczne w Java/Selenium - konfiguracja WebDriver

**Architektura**

Hierarchia `WebDriver` to zbiór klas służących do sterowania różnymi przeglądarkami internetowymi. Struktura ta została zaprojektowana zgodnie z paradygmatem programowania obiektowego. Na jej szczycie znajduje się interfejs WebDriver, który stanowi nadrzędny element całej hierarchii. Niżej znajdują się klasy w Javie, które odpowiadają za obsługę poszczególnych przeglądarek – na przykład, aby sterować lokalną przeglądarką Chrome, należy użyć instancji klasy ChromeDriver. Analogicznie, dla innych przeglądarek dostępne są odpowiednie klasy: EdgeDriver dla Microsoft Edge, FirefoxDriver dla Firefoksa, SafariDriver dla Safari, OperaDriver dla Opery oraz InternetExplorerDriver dla Internet Explorera. Do obsługi przeglądarek zdalnych służy klasa `RemoteWebDriver`.

## Driver

Aby utworzyć obiekt z hierarchii WebDriver (np. `ChromeDriver`), konieczne jest wcześniejsze rozwiązanie i skonfigurowanie odpowiadającego sterownika. Przykładowo, aby sterować przeglądarką Chrome za pomocą `ChromeDriver`, najpierw należy mieć zainstalowaną tę przeglądarkę na lokalnej maszynie, a następnie zadbać o właściwy sterownik (`chromedriver`). Ręczne zarządzanie sterownikami jest czasochłonne i podatne na błędy, dlatego zaleca się automatyzację tego procesu za pomocą narzędzi takich jak `WebDriverManager`.

**Zarządzanie sterownikami (Driver Managers):**

**Konieczność posiadania sterowników**: Sterowniki są niezbędne, aby `Selenium WebDriver` mogło kontrolować przeglądarki internetowe.

**Proces zarządzania sterownikami**: Obejmuje trzy główne etapy:

1. **Pobieranie**: Każda przeglądarka (np. Chrome, Firefox) wymaga specyficznego sterownika (np. `chromedriver`, `geckodriver`). Na przykład, używamy `chromedriver` do kontrolowania Chrome lub `geckodriver` dla Firefoxa. Sterownik jest plikiem binarnym specyficznym dla platformy. Dlatego musimy pobrać odpowiedni sterownik dla danego systemu operacyjnego (zazwyczaj Windows, macOS lub Linux). Dodatkowo, musimy wziąć pod uwagę wersję sterownika, ponieważ dana wersja sterownika jest kompatybilna z określoną wersją przeglądarki (lub zakresem wersji). Na przykład, aby używać Chrome 91.x, musimy pobrać `chromedriver` 91.0.4472.19. Zgodność przeglądarki ze sterownikiem zwykle znajdujemy w dokumentacji sterownika lub informacjach o wydaniu.
2. **Konfiguracja**: Pobrany sterownik musi zostać udostępniony skryptom `Selenium WebDriver`.
3. **Konserwacja**: Nowoczesne przeglądarki aktualizują się automatycznie i po cichu, bez pytania użytkownika. Z tego powodu, w kontekście `Selenium WebDriver`, musimy utrzymywać zgodność wersji przeglądarki i sterownika w czasie dla tych tzw. przeglądarek "evergreen" (wiecznie zielonych, tj. wiecznie aktualnych).

Proces konserwacji sterowników może być czasochłonny. Co więcej, może powodować problemy dla użytkowników Selenium WebDriver (np. nieudane testy z powodu niezgodności przeglądarki i sterownika po automatycznej aktualizacji przeglądarki). Z tego powodu, tak zwane menedżery sterowników (`driver managers`) mają na celu w pewnym stopniu zautomatyzowanie procesu zarządzania sterownikami.

`WebDriverManager` to biblioteka dla Javy, która automatycznie pobiera, konfiguruje i utrzymuje sterowniki do przeglądarek wymagane przez Selenium WebDriver. Po dodaniu tej biblioteki jako zależności do projektu (np. przez Maven lub Gradle), można wygodnie zarządzać sterownikami poprzez wywołania statycznych metod klasy `WebDriverManager`. Każda przeglądarka ma dedykowaną metodę, która automatyzuje cały proces - od wykrycia wersji przeglądarki, przez pobranie właściwego sterownika, aż po ustawienie zmiennych środowiskowych.

Analogiczne metody dostępne są dla innych przeglądarek. `WebDriverManager` udostępnia także bardziej zaawansowane opcje konfiguracji oraz może być używany jako narzędzie CLI, serwer REST czy agent Java.

Takie podejście znacząco upraszcza konfigurację środowiska testowego oraz minimalizuje ryzyko błędów związanych z niezgodnością wersji sterowników i przeglądarek.

### Instalacja driverów

Instalacja sterowników (`driver`) Selenium w Javie może odbywać się na kilka sposobów, w tym poprzez narzędzie Maven, manualne pobranie plików ZIP lub z wykorzystaniem biblioteki WebDriverManager, które zostało opisane wyżej. Poniżej zostały przedstawione bardziej klasyczne sposoby:

**Instalacja przy użyciu Maven**

Maven jest narzędziem służącym do budowania projektów i zarządzania zależnościami. Aby zainstalować biblioteki Selenium przy użyciu Maven, należy dodać odpowiednie zależności do pliku `pom.xml` projektu. Po dodaniu zależności, Maven automatycznie pobierze wymagane pliki JAR Selenium. Jest to rekomendowany sposób, ponieważ Maven sam zarządza pobieraniem i aktualizacją bibliotek.

**Kroki:**

1. Otwórz plik `pom.xml` swojego projektu Maven.
2. W sekcji `<dependencies>` dodaj wpis dla Selenium Java. Przykładowa zależność może wyglądać następująco:

```xml
<dependencies>
    <dependency>
        <groupId>org.seleniumhq.selenium</groupId>
        <artifactId>selenium-java</artifactId>
        <version>NAJNOWSZA_WERSJA</version> <!--Zastąp NAJNOWSZA_WERSJA numerwem wersji Selenium --> 
    </dependency>
</dependencies>

```

3. Zapisz plik `pom.xml`. Maven automatycznie pobierze i skonfiguruje biblioteki Selenium dla Twojego projektu.

**Instalacja przy użyciu plików ZIP (manualna)**

Ta metoda polega na ręcznym pobraniu plików JAR Selenium i dodaniu ich do ścieżki budowania projektu w środowisku programistycznym (IDE), takim jak Eclipse. **Kroki:**

1. **Pobierz pliki Selenium:** Wejdź na oficjalną stronę Selenium (selenium.dev) i pobierz pliki klienckie Selenium WebDriver dla Javy. Zazwyczaj są one dostępne w formie archiwum ZIP.
2. **Wypakuj pliki:** Rozpakuj pobrane archiwum ZIP do wybranego folderu na dysku twardym, na przykład `C:\\selenium-DRIVE`.
3. **Skonfiguruj projekt w IDE (np. Eclipse):**
    - Utwórz nowy projekt Java w Eclipse.
    - Kliknij prawym przyciskiem myszy na projekt i wybierz "Properties" (Właściwości).
    - Przejdź do sekcji "Java Build Path" (Ścieżka budowania Javy).
    - Wybierz zakładkę "Libraries" (Biblioteki).
    - Kliknij "Add External JARs..." (Dodaj zewnętrzne pliki JAR...).
    - Przejdź do folderu, w którym wypakowałeś pliki Selenium, i dodaj wszystkie pliki JAR (zarówno te w głównym folderze, jak i te w podfolderze `libs`).
    - Kliknij "Apply and Close" (Zastosuj i zamknij) lub "OK".

Po wykonaniu tych kroków, biblioteki Selenium będą dostępne w projekcie. Należy pamiętać, że oprócz bibliotek Selenium, konieczne jest również pobranie odpowiedniego sterownika dla przeglądarki, którą używamy (np. ChromeDriver dla Google Chrome, GeckoDriver dla Firefoxa). Ścieżkę do tego sterownika należy następnie wskazać w kodzie, np. za pomocą `System.setProperty("webdriver.chrome.driver", "/sciezka/do/chromedriver");`.

**Uruchomienie WebDriver poprzez System.property**

```java
System.setProperty("webdriver.chrome.driver","D:\\Programowanie\\Testy\\ChromeDriver\\chromedriver.exe");
WebDriver driver = new ChromeDriver();
```

**Instalacja przy użyciu WebDriverManager**

`WebDriverManager` to biblioteka Java typu open-source, która automatyzuje zarządzanie sterownikami wymaganymi przez Selenium WebDriver. Obejmuje to pobieranie, konfigurację i utrzymanie sterowników. Eliminuje to potrzebę manualnego pobierania sterowników przeglądarek i konfigurowania ścieżek systemowych.

**Kroki:**

1. **Dodaj zależność WebDriverManager do projektu:** Jeśli używamy Maven, dodajemy następującą zależność do pliku `pom.xml`:

```xml
<dependency>
    <groupId>io.github.bonigarcia</groupId>
    <artifactId>webdrivermanager</artifactId>
    <version>NAJNOWSZA_WERSJA_WDM</version> <!--Zastąp NAJNOWSZA_WERSJA numerem wersji Selenium --> 
</dependency>

```

2. **Użyj WebDriverManager w kodzie:** W swoim kodzie Java, przed inicjalizacją instancji `WebDriver`, wywołujemy odpowiednią metodę `setup()` z WebDriverManager dla przeglądarki, której użyjemy. Przykład dla Chrome:

```java
import io.github.bonigarcia.wdm.WebDriverManager;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;

public class MySeleniumTest {
    public static void main(String[] args) {
        WebDriverManager.chromedriver().setup();
        WebDriver driver = new ChromeDriver();
        // dalszy kod testu
        driver.quit();
    }
}
```

| Wywołanie WebDriverManager | Przeglądarka | Sterownik |
| --- | --- | --- |
| WebDriverManager.chromedriver().setup(); | Chrome | chromedriver |
| WebDriverManager.edgedriver().setup(); | Edge | msedgedriver |
| WebDriverManager.firefoxdriver().setup(); | Firefox | geckodriver |

`WebDriverManager` automatycznie wykryje wersję przeglądarki zainstalowanej w systemie, pobierze odpowiednią wersję sterownika i skonfiguruje niezbędne właściwości systemowe Javy. Biblioteka ta pomaga również w utrzymaniu kompatybilności między sterownikiem a przeglądarką, co jest istotne, ponieważ przeglądarki często się automatycznie aktualizują.

## **WebDriver Builder /** RemoteWebDriver

API Selenium WebDriver udostępnia klasę `RemoteWebDriver` do sterowania zdalnymi przeglądarkami. Architektonicznie klasa ta jest rodzicem pozostałych klas WebDriver, które sterują lokalnymi przeglądarkami (tj. ChromeDriver, FirefoxDriver itp.). W ten sposób można korzystać z obiektów RemoteWebDriver w taki sam sposób, jak z WebDriver. `RemoteWebDriver` to klasa w bibliotece Selenium, która umożliwia automatyzację przeglądarek internetowych na zdalnych maszynach. Jest to szczególnie przydatne w scenariuszach, gdzie testy muszą być uruchamiane na różnych systemach operacyjnych i konfiguracjach przeglądarek, co pozwala na szerokie testowanie aplikacji w różnych środowiskach.

- Określić adres URL zdalnej maszyny (np. adres Selenium Grid).
- Skonfigurować opcje przeglądarki za pomocą odpowiedniej klasy (np. ChromeOptions dla Google Chrome).
- Utworzyć instancję RemoteWebDriver, przekazując adres URL oraz opcje przeglądarki.

[WebDriver Builder](https://www.selenium.dev/selenium/docs/api/java/) to wzorzec projektowy oraz klasa (np. **`Builder`** w Selenium) pochodząca od `RemoteWebDriver`, która umożliwia elastyczne i wygodne konfigurowanie oraz tworzenie instancji WebDrivera, czyli sterownika automatyzującego przeglądarkę internetową. Dzięki zastosowaniu wzorca `builder`, użytkownik może w sposób czytelny ustawiać różne opcje, takie jak wybór przeglądarki, adres serwera zdalnego, dodatkowe możliwości (capabilities), metadane czy konfiguracje połączenia, a następnie jednym wywołaniem metody **`build()`** utworzyć gotowy obiekt WebDrivera, zgodny z wybraną konfiguracją. Może to zrobić poprzez sieć lub w chmurze. `RemoteWebDriver` dziedziczy po `WebDriver`. Używamy gdy chcemy uruchamiać testy na zdalnym serwerze (np. Selenium Grid, Docker, BrowserStack, Selenoid) (da się też lokalnie).

Metody WebDriver Builder:

| Metoda | Opis |
| --- | --- |
| `oneOf(Capabilities options)` | Przeglądarko-specyficzne możliwości (capabilities) |
| `addAlternative(Capabilities options)` | Alternatywne możliwości specyficzne dla przeglądarki |
| `addMetadata(String key, Object value)` | Dodaj niestandardowe metadane, zwykle używane do żądania dodatkowych funkcji u dostawców chmurowych  |
| `setCapability(String capabilityName, Object value)` | Indywidualne możliwości specyficzne dla przeglądarki, np. `.setCapability("acceptInsecureCerts", true)`
`.setCapability("pageLoadStrategy", "eager")`  |
| `address(String uri) / address(URL url) / address(URI uri)` | Ustaw adres zdalnego serwera. |
| `config(ClientConfig config)` | Szczegółowa konfiguracja przy korzystaniu ze zdalnego serwera, np. timeout połączenia lub ustawienia proxy |
| `withDriverService(DriverService service)` | Szczegółowa konfiguracja lokalnego sterownika (np. chromedrivera), taka jak lokalizacja pliku, używany port, timeout lub argumenty. |
| `build()` | Ostatnia metoda we wzorcu buildera, służąca do utworzenia instancji WebDrivera |

Istnieją różne sposoby tworzenia instancji obiektu `RemoteWebDriver`, tzn. drivera który uruchamiany jest na innej maszynie, np. serwerze. To klasyczny, starszy, ale bardzo stabilny i wciąż oficjalnie wspierany sposób tworzenia `RemoteWebDrivera`. - WebSocket - co to jest?
    
    ### Czym są WebSockety?
    
    WebSocket to protokół komunikacyjny działający przez stałe, dwukierunkowe połączenie TCP między klientem (np. przeglądarką) a serwerem. Pozwala na szybki i efektywny przesył danych w czasie rzeczywistym — bez potrzeby ciągłego odpytywania serwera (jak w przypadku HTTP). Połączenie **zawsze zaczyna się jako HTTP, a po** handshake: klient i serwer mogą wysyłać wiadomości **niezależnie,** brak request/response dane lecą w **ramkach (frames).** 
    
    - WebSocket = **stałe, dwukierunkowe połączenie**
    - rozwiązuje problem real-time
    - zaczyna się od HTTP, potem zmienia protokół
        
        
        | Cecha | HTTP | WebSocket |
        | --- | --- | --- |
        | Kierunek | Jednokierunkowy | Dwukierunkowy |
        | Połączenie | Krótkotrwałe | Długotrwałe |
        | Inicjator | Klient | Klient i serwer |
        | Latencja | Wyższa | Niska |
        | Skalowanie | Łatwe | Trudniejsze |
    
    ---
    
    ### WebSockety w kontekście Selenium i komunikacji z przeglądarką
    
    - **Selenium 4 i późniejsze wersje** korzystają z nowoczesnych protokołów do komunikacji z przeglądarkami. Zamiast starego JSON Wire Protocol, od wersji 4 w Selenium jest wsparcie dla WebDrivera w nowym standardzie W3C, a także integracja z **CDP (Chrome DevTools Protocol)** — które opiera się na WebSocketach.
    - **CDP** (głównie w Chrome/Edge) to protokół, który pozwala narzędziom automatyzującym (jak Selenium) kontrolować przeglądarkę na bardzo niskim poziomie — m.in. wykonywać JavaScript, zbierać logi, monitorować sieć, manipulować DOM, robić screenshoty i wiele więcej.
    - Komunikacja między klientem Selenium a przeglądarką często odbywa się właśnie przez WebSocket, zwłaszcza gdy korzystasz z RemoteWebDriver i chcesz sterować przeglądarką zdalnie (np. w Selenium Grid, czy w kontenerze Docker).
    - WebSocket zapewnia stałe połączenie, więc Selenium może wysyłać komendy i natychmiast odbierać odpowiedzi, bez konieczności ponownego nawiązywania połączenia.
    
    ---
    
    ### Dlaczego czasem pojawiają się błędy z WebSocketami w Selenium?
    
    - Jeśli np. Selenium próbuje nawiązać połączenie WebSocket z przeglądarką lub serwerem Selenium Grid, ale coś blokuje połączenie (firewall, złe adresy, niezgodne wersje), wtedy mogą się pojawić błędy typu `ConnectionFailedException: JdkWebSocket initial request execution error`.
    - Także, jeśli wersja Javy lub klienta Selenium jest niezgodna z wersją serwera lub przeglądarki, mogą pojawić się problemy z WebSocketami.
</aside>

## **Augmenter**

`RemoteWebDriver` działa jako podstawowa klasa dla interakcji z przeglądarką, ale domyślnie nie implementuje wszystkich dostępnych interfejsów Selenium. Gdy używamy RemoteWebDriver, niektóre funkcje, jak zrzuty ekranu, nie działają bez augmentacji. Na przykład funkcje takie jak robienie zrzutów ekranu (`TakesScreenshot`) lub zarządzanie plikami mogą wymagać dodatkowych funkcjonalności wprowadzanych przez różne sterowniki. W testach rozproszonych (np. Selenium Grid), gdzie `RemoteWebDriver` nie posiada pełnej implementacji metod lokalnego WebDrivera. Dlatego powstała klasa Augmenter, która dynamicznie "rozszerza" instancję `RemoteWebDriver` o implementacje interfejsów, które są zgodne z jego możliwościami (`Capabilities`). Automatycznie sprawdza, jakie interfejsy mogą być zaimplementowane przez dany sterownik i umożliwia ich użycie w zdalnej przeglądarce.

```java
public class AugmenterExample {
    public static void main(String[] args) throws Exception {
        // Konfiguracja przeglądarki
        ChromeOptions options = new ChromeOptions();
        URL gridUrl = new URL("http://adres-zdalnej-maszyny:4444/wd/hub");
        // Utworzenie instancji RemoteWebDriver
        RemoteWebDriver remoteWebDriver = new RemoteWebDriver(gridUrl, options);
        // Rozszerzenie funkcjonalności za pomocą Augmentera
        WebDriver augmentedDriver = new Augmenter().augment(remoteWebDriver);
        // Nawigacja do strony
        augmentedDriver.get("https://example.com");
        // Jeśli sterownik wspiera TakesScreenshot, zrób zrzut ekranu
        if (augmentedDriver instanceof TakesScreenshot) {
            File screenshot = ((TakesScreenshot) augmentedDriver).getScreenshotAs(OutputType.FILE);
            FileUtils.copyFile(screenshot, new File("screenshot.png"));
            System.out.println("Zrzut ekranu zapisano jako screenshot.png");
        } else {
            System.out.println("Sterownik nie wspiera robienia zrzutów ekranu.");
        }
        // Zamknięcie przeglądarki
        augmentedDriver.quit();
    }
}
```

## Thready driverów

Thready, czyli wątki, są funkcjonalnością Javy pozwalającą na uruchamianiu tej samej aplikacji korzystającej z tych samych plików, zmiennych itp. wiele razy, bez wprowadzania kkolizji wynikających z równoległego wykrozystania tego samego kodu. Jest to przydatne także do oddzielenia od siebie testów, szczególnie gdy uruchamiane są one równolegle (paralllel). Nie jest to funkcjonalność Selenium, lecz należy do funkcjonalności Javy, jednak jest bardzo przydatna przy projektowaniu i uruchamiania testów. 

Wątki (ang. *threads*) są mechanizmem Javy umożliwiającym **współbieżne wykonywanie wielu fragmentów kodu w ramach jednej aplikacji (jednego procesu JVM)**. Oznacza to, że kilka zadań może być realizowanych równolegle lub pozornie równolegle, przy **współdzieleniu tych samych zasobów**, takich jak pamięć, zmienne czy pliki.

Java zapewnia rozbudowane mechanizmy kontroli współbieżności (np. `synchronized`, `Lock`, `volatile`, klasy z pakietu `java.util.concurrent`), które pozwalają **zapobiegać kolizjom**, takim jak: race condition, niespójność danych, deadlock.

Dzięki temu możliwe jest bezpieczne korzystanie z tego samego kodu i zasobów przez wiele wątków jednocześnie.

Wątki są szczególnie przydatne w **testach automatycznych uruchamianych równolegle (parallel execution)**, np. gdy: kilka testów jest wykonywanych jednocześnie, testy korzystają z tych samych klas pomocniczych, każdy test powinien działać na własnym kontekście (np. osobna instancja WebDrivera). W takim przypadku kluczowe jest: **oddzielenie stanu testów** (np. przez osobne obiekty, `ThreadLocal`, osobne sesje przeglądarki), unikanie współdzielonych, modyfikowalnych danych statycznych.

**💡Warto pamiętać, że mechanizm wątków nie jest częścią Selenium, jest elementem platformy Java.**

Selenium nie zarządza wątkami, ale działa w środowisku wielowątkowym, jeśli testy są uruchamiane równolegle (np. przez TestNG, JUnit, Gradle, Maven Surefire). Dlatego znajomość wątków i zasad współbieżności w Javie jest **niezbędna do poprawnego projektowania testów równoległych w Selenium**.

```java
	private static final ThreadLocal<WebDriver> DRIVER_THREAD = new ThreadLocal<>();
	(...) //Tworzymy driver
	DRIVER_THREAD.set(driver);
	
//W przypadku gdy chcemy odwołać się do takiego drivera w teście, robimy to poprzez GET.
public static WebDriver getDriver() {
		WebDriver driver = DRIVER_THREAD.get();
		return driver;
}

public static void quitDriver() {
    if (DRIVER_THREAD.get() != null) {
        DRIVER_THREAD.get().quit(); // Zamyka przeglądarkę
        DRIVER_THREAD.remove();     // Czyści zmienną w wątku, zapobiega memory leaks
}
```

Przeglądarki takie jak **Chrome, Edge (Chromium)** oraz **Firefox** w kontekście Selenium/WebDrivera wykorzystują **trzy logiczne poziomy konfiguracji**, z których każdy działa na innym etapie inicjalizacji i służy innemu celowi:

1. **Capabilities (standard W3C)**
2. **Argumenty uruchomienia przeglądarki (CLI flags)**
3. **Preferencje profilu użytkownika (prefs / profile settings)**

Każdy z tych mechanizmów ma inne zastosowanie i inny wpływ na stabilność testów.

**1. Ustawienia standardowe – W3C Capabilities**

To **fundament sesji WebDrivera**, zgodny ze specyfikacją W3C. Capabilities opisują **jaką przeglądarkę i na jakim środowisku** uruchamiamy. Celem ich jest:

- Deklaratywne opisanie środowiska testowego.
- Zapewnienie kompatybilności między Selenium, Gridem i usługami chmurowymi.

**Kluczowe capability**

- `browserName`
- `browserVersion`
- `platformName`
- `acceptInsecureCerts` – akceptacja certyfikatów SSL

**Charakterystyka**

- Odczytywane **przed uruchomieniem przeglądarki**
- Niezależne od konkretnego silnika (Chrome / Firefox / Edge)

---

**2. Argumenty uruchomienia przeglądarki (`addArguments`)**

Argumenty przekazywane **bezpośrednio do procesu przeglądarki** (np. `chrome.exe`) w momencie jej startu. Celem ich jest:

- Zmiana sposobu działania **procesu przeglądarki**
- Kontrola trybu pracy i zasobów systemowych

**Charakterystyka**

- Są interpretowane **przy starcie procesu**
- Działają **zanim zostanie załadowany profil użytkownika**
- Odpowiadają flagom CLI używanym przy ręcznym uruchamianiu Chrome

**Przykłady praktyczne**

- `-headless` – uruchomienie bez GUI (CI/CD)
- `-incognito` – start z czystą sesją
- `-disable-gpu` – stabilność w środowiskach bez akceleracji
- `-proxy-server=host:port` – routing ruchu sieciowego
- `-use-fake-ui-for-media-stream` – automatyczna zgoda na media

---

**3. Preferencje użytkownika – `prefs` (ustawienia profilu)**

Najbardziej szczegółowy poziom konfiguracji, odpowiadający **ustawieniom zapisywanym w profilu użytkownika przeglądarki**.  Celem ich jest:

- Kontrola **zachowania przeglądarki wewnątrz profilu**
- Symulują zachowanie użytkownika i konfigurację przeglądarki (jakbyś ustawił coś w `chrome://settings`).

**Charakterystyka**

- Przekazywane jako **mapa key–value**
- Technicznie ustawiane przez `setExperimentalOption("prefs", map)`
- Obowiązują **po załadowaniu profilu**, ale przed interakcją z aplikacją

**Przykłady biznesowe**

- `download.default_directory` – kontrola lokalizacji raportów
- `profile.default_content_setting_values.notifications` – blokada powiadomień
- `credentials_enable_service` – wyłączenie zapisu haseł
- `profile.default_content_setting_values.geolocation` – kontrola geolokalizacji

> 💡Uwaga: nazwa “experimental options” wynika z API Selenium – same prefs nie są eksperymentalne, tylko są zapisywane w profilu.
> 

---

Porównanie techniczne:

| Cecha | 1. Capabilities (W3C) | 2. Argumenty (CLI) | 3. Preferencje (Prefs) |

| `getRect()` | Pobiera prostokąt (rectangle) elementu, zawierający lokalizację i rozmiar. |
| `getSize()` | Pobiera rozmiar elementu (szerokość i wysokość w pikselach). |
| `getTagName()` | Pobiera nazwę znacznika (tag name) elementu, np. `input` lub `div`. |
| `getText()` | Pobiera widoczny tekst wewnątrz elementu. |
| `isDisplayed()` | Sprawdza, czy element jest widoczny na stronie. |
| `isEnabled()` | Sprawdza, czy element jest aktywny (enabled). |
| `isSelected()` | Sprawdza, czy element (np. checkbox) jest zaznaczony. |
| `sendKeys(CharSequence... keysToSend)` | Wysyłanie tekstu (symulowanie pisania) do danego elementu. |
| `submit()` | Wysyła formularz, w którym znajduje się element. |

**🔍 Różnice między metodami get…**

| Metoda | Co zwraca? |
| --- | --- |
| `getAttribute("value")` | ✅ Wartość atrybutu widziana przez **JavaScript + DOM + HTML** – najczęściej używana, ale może zwracać **przetworzone** dane. To metoda "starego typu". Próbuje być inteligentna: najpierw sprawdza stan obecny (property), a jeśli go nie ma, patrzy w kod HTML (attribute). Problem polega na tym, że w nowoczesnych aplikacjach (React, Angular, Vue) stan strony zmienia się tak szybko, że `getAttribute` może zwrócić nieprzewidywalny wynik. **Dlatego nowe Selenium wprowadziło rozdzielenie na dwie precyzyjne metody.** |
| `getDomAttribute("value")` | ✅ **Atrybut z oryginalnego HTML** – tak jak został zapisany w znaczniku. Nie uwzględnia zmian wykonanych przez JS czy użytkownika (zmiany wartości inputu). |
| `getDomProperty("value")` | ✅ **Właściwość DOM** – aktualna wartość w obiekcie JavaScript. To, co JS „widzi” w `element.value`, `element.checked`, itp. |

## Mechanizm przesyłania plików

Standardowy mechanizm przesyłania plików w aplikacjach internetowych wykorzystuje elementy `<input>` z atrybutem `type="file"`. Na przykład, formularz testowy może zawierać taki element:

```html
<input class="form-control" type="file" name="my-file">
```

**Dlaczego potrzebna jest symulacja klawiatury?**

API `Selenium WebDriver` **nie zapewnia wbudowanego mechanizmu** do obsługi pól typu `<file>`. Zamiast tego należy traktować elementy `<input>` do przesyłania plików jak zwykłe pola tekstowe i **symulować wprowadzanie ścieżki do pliku przez użytkownika**. W praktyce oznacza to:

1. Znalezienie elementu `<input type="file">`.
2. Wysłanie do niego **bezwzględnej ścieżki** (pełnego adresu) pliku, który ma zostać przesłany, np.:

```python
element.send_keys("/ścieżka/do/pliku/testowy_plik.txt")
```

**Kluczowe uwagi**

- Ścieżka musi być **bezwzględna** (np. `C:\\Users\\user\\Desktop\\file.txt` w systemie Windows lub `/home/user/file.txt` w Linux).
- Mechanizm ten działa tylko dla elementów `<input type="file">`.
- Nie jest wymagane otwieranie okien dialogowych systemowych – Selenium bezpośrednio "wstrzykuje" ścieżkę do pola.

### Przesyłanie plików do zdalnej przeglądarki w Selenium

Podczas przesyłania pliku do **zdalnej przeglądarki**, musimy jawnie wskazać, że plik pochodzi z **lokalnego systemu plików**. W tym celu używa się klasy `LocalFileDetector`.

**Konfiguracja detektora plików lokalnych**

Następująca linia kodu pokazuje, jak ustawić detektor plików lokalnych:

```java
((RemoteWebDriver) driver).setFileDetector(new LocalFileDetector());

```

**Wyjaśnienie:**

- `RemoteWebDriver` – rzutowanie drivera na typ zdalny (wymagane, by użyć metod specyficznych dla zdalnych połączeń).
- `LocalFileDetector` – klasa odpowiedzialna za wykrywanie plików z lokalnego systemu i automatyczne przesyłanie ich do zdalnego środowiska.

**Przykład użycia**

```java
if (DriverFactory.getDriver() instanceof RemoteWebDriver) {
            ((RemoteWebDriver) DriverFactory.getDriver()).setFileDetector(new LocalFileDetector());
        }
fileField.sendKeys(Path.of(path).toAbsolutePath().toString());

WebElement fileInput = driver.findElement(By.cssSelector("input[type='file']"));
fileInput.sendKeys("/ścieżka/do/lokalnego/pliku.txt"); // Ścieżka lokalna
```

💡**Uwaga:** Bez ustawienia `LocalFileDetector`, zdalny WebDriver nie będzie w stanie odnaleźć pliku na lokalnym dysku!

    Dawniej (WebDriver klasyczny)

    - Konfiguracja odbywała się wyłącznie przez preferencje przeglądarki
    - Test ręcznie sprawdzał katalog systemowy

        - BiDi

        - komendy Dockera
    
    ---
    
    **5. Finalne rozwiązanie – strategia hybrydowa**
    
    Najbardziej stabilne podejście:
    
    1. Najpierw próba przez oficjalne API Selenium
    2. W razie błędu – fallback do klasycznego sprawdzania folderu
    3. Dla środowiska lokalnego – wyłącznie FS

    **6. Wnioski architektoniczne**
    
    1. **BiDi zmienia filozofię downloadów**, ale Grid nie zawsze wspiera ją w 100%
    2. Chrome Node zachowuje się inaczej niż Edge
    3. Najbezpieczniejszy model to API → fallback to filesystem
    4. Test Selenium nie powinien:
    - wywoływać dockera
    - znać nazw kontenerów
    - opierać się na wewnętrznej strukturze Grid

## Interakcja z suwakami zakresu (**`<input type="range">`**) w Selenium

Podobnie jak w przypadku przesyłania plików, elementy **`<input type="range">`** (tzw. **suwaki zakresu**) wymagają specjalnego podejścia w Selenium WebDriver. Przykładowy element z formularza testowego:

`<input type="range" class="form-range" name="my-range" min="0" max="10" step="1" value="5">`

**Problem**

Selenium WebDriver **nie oferuje wbudowanej metody** do manipulowania suwakami. Te elementy pozwalają użytkownikom wybrać wartość z zakresu za pomocą graficznego suwaka, ale nie da się ich bezpośrednio przeciągnąć.

**Rozwiązanie**

Można symulować akcje klawiatury, aby ustawić żądaną wartość tekstowo:

`javaWebElement slider = driver.findElement(By.name("my-range"));
slider.sendKeys("8"); *// Ustawia wartość na 8 (w zakresie 0-10)*`

**Dlaczego to działa?**

Mimo że suwak jest elementem graficznym, Selenium traktuje go jak zwykłe pole tekstowe. Wysłanie wartości za pomocą **`sendKeys()`** aktualizuje właściwość **`value`** suwaka, co symuluje interakcję użytkownika.

**Uwaga:** Wartość musi być w formacie tekstowym i mieścić się w zakresie zdefiniowanym przez **`min`** i **`max`**!

## Dropdown Lists

Typowym elementem formularzy internetowych są listy rozwijane. Pola te pozwalają użytkownikom wybrać jeden lub więcej elementów z listy opcji. Klasycznymi znacznikami HTML używanymi do renderowania tych pól są `<select>` i `<options>`.

Element `<datalist>` służy do tworzenia listy podpowiedzi dla pola tekstowego (`<input>`). Umożliwia użytkownikowi:

- wybranie **jednej z sugerowanych opcji** z listy rozwijanej,
- **wprowadzenie własnej, niestandardowej wartości**.

**🆚 Różnica między `<select>` a `<datalist>`**

| Cecha | `<select>` | `<datalist>` |
| --- | --- | --- |
| Typ pola | Gotowa lista wyboru | Pole tekstowe z podpowiedziami |
| Wprowadzanie własnej wartości | ❌ Nie | ✅ Tak |
| Typ HTML elementu | `<select>` + `<option>` | `<input list="id">` + `<datalist>` |
| Wsparcie w Selenium | Tak, przez klasę `Select` | Nie, interakcja jak z polem tekstowym |

Selenium **nie posiada dedykowanej klasy `Select`** dla `<datalist>`, więc musimy traktować to jak **zwykłe pole tekstowe**. Możemy pobrać wartość z jednej z opcji i wpisać je w pole.

```java
datalist.click();
WebElement option = driver
.findElement(By.xpath("//datalist/option[2]"));
String optionValue = option.getAttribute("value");
datalist.sendKeys(optionValue);
```

## Działania za pomocą myszki

Oprócz klawiatury, drugim podstawowym urządzeniem wejściowym wykorzystywanym do interakcji z aplikacjami internetowymi jest **mysz komputerowa**. Przede wszystkim pojedyncze kliknięcie (znane również jako kliknięcie lewym przyciskiem myszy lub po prostu kliknięcie) jest symulowane przez API Selenium WebDriver za pomocą metody **`click()`**, która jest jedną z metod dostępnych dla każdego obiektu **`WebElement`** w Selenium WebDriver.

Typowe przypadki użycia tej funkcji:

- nawigacja po stronach internetowych
- interakcja z polami wyboru (checkbox) i przyciskami opcji (radio) w formularzach internetowych
- kliknięcie prawym przyciskiem myszy (tzw. kliknięcie kontekstowe)
- podwójne kliknięcie
- przesuwanie kursora
- przeciąganie i upuszczanie
- najechanie kursorem (mouseover)

Selenium WebDriver umożliwia symulowanie tych akcji za pomocą klasy pomocniczej o nazwie **Actions**. 

```java
Actions action = new Actions(driver);
actions(webElement).akcje().build().perform();
```

Selenium WebDriver udostępnia klasę [**Actions**](https://www.selenium.dev/selenium/docs/api/java/org/openqa/selenium/interactions/Actions.html), która jest potężnym narzędziem do automatyzacji różnych akcji użytkownika – zarówno związanych z klawiaturą, jak i myszą. Klasa ta wykorzystuje wzorzec projektowy **builder**. Dzięki temu możesz łączyć (łańcuchować) kilka metod (czyli różnych akcji), a następnie wykonać je wszystkie na końcu, wywołując metodę **`build()`**. Jeśli nie budujemy łańcucha, wystarczy samo `perform**()`.**

| Metoda | Opis |
| --- | --- |
| `keyDown(CharSequence key)`
`keyDown(WebElement target, CharSequence key)` | Wysyła pojedynczy klawisz (może to być znak specjalny z klasy Keys) w bieżącej pozycji (lub na wskazanym elemencie). Klawisz pozostaje wciśnięty do wywołania `keyUp()`. |
| `keyUp(CharSequence key)`
 `keyUp(WebElement target, CharSequence key)` | Zwalnia klawisz wcześniej wciśnięty metodą `keyDown()`. |
| `sendKeys(CharSequence... keys)`
`sendKeys(WebElement target, CharSequence... keys)` | Wysyła sekwencję klawiszy w bieżącej pozycji (lub na wskazanym elemencie). Różni się od `WebElement#sendKeys` tym, że: 1) klawisze modyfikujące (np. CONTROL, SHIFT) nie są zwalniane automatycznie, 2) nie następuje ponowne ustawienie fokusu na elemencie, więc np. TAB działa prawidłowo. |
| `clickAndHold()`
`clickAndHold(WebElement target)` | Kliknięcie (przytrzymanie) bez zwalniania w bieżącej pozycji (lub na środku wskazanego elementu). |
| `release()`
`release(WebElement target)` | Zwalnia lewy przycisk myszy wcześniej wciśnięty metodą `clickAndHold()`. |
| `click()`
`click(WebElement target)` | Kliknięcie w bieżącej pozycji (lub na wskazanym elemencie). |
| `doubleClick()`
`doubleClick(WebElement target)` | Podwójne kliknięcie w bieżącej pozycji (lub na wskazanym elemencie). |
| `contextClick()`
`contextClick(WebElement target)` | Kliknięcie prawym przyciskiem myszy (klik kontekstowy) w bieżącej pozycji (lub na wskazanym elemencie). |
| `moveToElement(WebElement target)`
`moveToElement(WebElement target, int xOffset, int yOffset)` | Przesuwa kursor myszy na środek (lub z przesunięciem) wskazanego elementu. |
| `moveByOffset(int xOffset, int yOffset)` | Przesuwa kursor myszy z bieżącej pozycji (domyślnie 0,0) o podane przesunięcie. |
| `dragAndDrop(WebElement source, WebElement target)` `dragAndDropBy(WebElement source, int xOffset, int yOffset)` | Akcja składa się z trzech kroków: 1) kliknięcie i przytrzymanie na środku (lub z przesunięciem) elementu źródłowego, 2) przesunięcie myszy do elementu docelowego, 3) zwolnienie przycisku myszy. |
| `pause(long pause)`
`pause(Duration duration)` | Wprowadza pauzę w łańcuchu akcji (w milisekundach lub jako obiekt Duration). |
| `build()` | Tworzy złożoną akcję zawierającą wszystkie poprzednie akcje. |
| `perform()` | Wykonuje akcję. |

# Strategie wyszukiwania elementów

Istotną umiejętnością pracy z aplikacjami przeglądarkowymi jest lokalizowanie elementów strony by wchodzić z nimi w interakcję. By to zrobić warto poznać strategie i mechanizmy wyszukiwania elementów.

| Lokalizator | Znajduje elementy na podstawie | Opis |
| --- | --- | --- |
| Nazwa tagu | Nazwy tagu HTML (np. a, p, div, img) | Wyszukuje elementy na podstawie nazwy tagu HTML, np. wszystkie `<div>`, `<a>`, itp. |
| Tekst linku | Dokładnego tekstu wyświetlanego w linku | Wyszukuje link (`<a>`) na podstawie dokładnego tekstu widocznego dla użytkownika. |
| Częściowy tekst linku | Fragmentu tekstu zawartego w linku | Wyszukuje link (`<a>`) na podstawie fragmentu tekstu – przydatne, gdy tekst jest długi lub zmienny. |
| Nazwa | Wartości atrybutu name | Wyszukuje elementy po atrybucie `name`, często stosowane w formularzach. |
| Id | Wartości atrybutu id | Wyszukuje elementy po unikalnym atrybucie `id`, najszybszy i najbardziej niezawodny sposób. |
| Nazwa klasy | Wartości atrybutu class | Wyszukuje elementy po nazwie klasy CSS, przydatne do zaznaczania grup elementów. |
| Selektor CSS | Wzorca CSS zgodnego z W3C Selectors | Umożliwia wyszukiwanie elementów za pomocą selektorów CSS, tak jak przy stylowaniu - bardzo elastyczne narzędzie. |
| XPath | Zapytania w języku XPath | Pozwala na wyszukiwanie elementów na podstawie ścieżek XPath, bardzo elastyczne, obsługuje złożone kryteria. |

**Objaśnienia:**

- *Nazwa tagu* – np. `driver.findElement(By.tagName("input"))` znajdzie wszystkie pola typu input.
- *Tekst linku* – np. `driver.findElement(By.linkText("Kontakt"))` znajdzie link o dokładnym tekście "Kontakt".
- *Częściowy tekst linku* – np. `driver.findElement(By.partialLinkText("Kon"))` znajdzie link zawierający fragment "Kon".
- *Nazwa* – np. `driver.findElement(By.name("email"))` znajdzie pole o nazwie "email".
- *Id* – np. `driver.findElement(By.id("submit"))` znajdzie element o id "submit".
- *Nazwa klasy* – np. `driver.findElement(By.className("btn-primary"))` znajdzie element(y) o klasie "btn-primary".
- *Selektor CSS* – np. `driver.findElement(By.cssSelector("div.menu > a.active"))` pozwala na bardzo precyzyjne wybieranie elementów.
- *XPath* – np. `driver.findElement(By.xpath("//div[@class='menu']//a[text()='Start']"))` umożliwia wyszukiwanie na podstawie struktury dokumentu XML/HTML.

## **Lokatory standardowe**

| **Lokator** | **Opis** |
| --- | --- |
| `By.id(String id)` | Znajduje element według wartości atrybutu `id`. |
| `By.name(String name)` | Znajduje element według wartości atrybutu `name`. |
| `By.className(String className)` | Znajduje element według wartości atrybutu `class`. |
| `By.tagName(String tagName)` | Znajduje element według nazwy znacznika HTML, np. `input`, `div`. |
| `By.linkText(String text)` | Znajduje element według pełnego tekstu linku (dla tagu `<a>`). |
| `By.partialLinkText(String text)` | Znajduje element według częściowego tekstu linku. |
| `By.xpath(String xpath)` | Znajduje element według wyrażenia XPath. |
| `By.cssSelector(String cssSelector)` | Znajduje element według selektora CSS. |

## **CSS Selector**

| Kategoria | Składnia | Opis po polsku | Przykład | Wyjaśnienie przykładu |
| --- | --- | --- | --- | --- |
| Uniwersalny | * | Wybiera wszystkie elementy | `*` | Dopasowuje wszystkie elementy na stronie |
| Typ | elementName | Wybiera wszystkie elementy o podanej nazwie tagu | `input` | Dopasowuje wszystkie elementy `<input>` |

## **XPath**

Lokator pozwalający na opisanie całej ścieżki do elementu (całej struktury DOM) lub tylko jej części. (Dokumentacja: [https://devhints.io/xpath](https://devhints.io/xpath)) 

| **Syntax** | **Opis** | **Przykład** | **Wyjaśnienie** |
| --- | --- | --- | --- |
| `/` | Wybiera elementy bezpośrednio od korzenia dokumentu | `/html/body/div` | Wybiera element `<div>` w strukturze od korzenia dokumentu HTML. |
| `//` | Wybiera elementy w dowolnym miejscu w dokumencie | `//div` | Wybiera wszystkie elementy `<div>` w dokumencie, niezależnie od ich lokalizacji. |
| `.` | Wskazuje na bieżący węzeł | `./span` | Wybiera element `<span>` będący dzieckiem bieżącego węzła. |
| `..` | Wskazuje na rodzica bieżącego węzła | `../div` | Wybiera element `<div>`, który jest rodzicem bieżącego elementu. |
| `@attribute` | Odwołuje się do wartości atrybutu | `//input[@type="text"]` | Wybiera wszystkie `<input>`, gdzie atrybut `type` ma wartość `text`. |
| `*` | Wybiera wszystkie elementy | `//div/*` | Wybiera wszystkie elementy będące dziećmi `<div>`. |
| `[condition]` | Wybiera elementy spełniające określony warunek | `//div[@class="header"]` | Wybiera wszystkie `<div>` z atrybutem `class="header"`. |
| `[n]` | Wybiera n-ty element węzła | `(//ul/li)[1]` | Wybiera pierwszy element `<li>` w strukturze `<ul>`. |
| `text()` | Wybiera tekst zawarty w węźle | `//h1[text()="Hello"]` | Wybiera element `<h1>`, który zawiera dokładny tekst `Hello`. |
| `contains()` | Sprawdza, czy ciąg tekstowy lub wartość atrybutu zawiera dany podciąg | `//a[contains(@href, "example")]` | Wybiera wszystkie linki (`<a>`) z atrybutem `href` zawierającym `example`. |
| `starts-with()` | Sprawdza, czy ciąg tekstowy lub wartość atrybutu zaczyna się od podciągu | `//input[starts-with(@id, "user")]` | Wybiera wszystkie `<input>`, których `id` zaczyna się od `user`. |
| `last()` | Wybiera ostatni element węzła | `//ul/li[last()]` | Wybiera ostatni element `<li>` w strukturze `<ul>`. |
| `position()` | Wybiera element na podstawie jego pozycji | `//tr[position()=2]` | Wybiera drugi wiersz `<tr>` w tabeli. |
| `and` | Łączy warunki za pomocą operatora logicznego **i** | `//input[@type="text" and @name="email"]` | Wybiera `<input>`, gdzie `type="text"` i `name="email"`. |
| `or` | Łączy warunki za pomocą operatora logicznego **lub** | `//input[@type="text" or @type="password"]` | Wybiera `<input>`, gdzie `type="text"` lub `type="password"`. |
| `|` | Łączy dwa różne selektory | `//div | //span` | Wybiera wszystkie elementy `div` oraz `span` |
| `ancestor::` | Wybiera wszystkich przodków bieżącego elementu | `//span/ancestor::div` | Wybiera wszystkie elementy `<div>`, które są przodkami `<span>`. |
| `descendant::` | Wybiera wszystkich potomków bieżącego elementu | `//div/descendant::p` | Wybiera wszystkie elementy `<p>`, które są potomkami `<div>`. |
| `following-sibling::` | Wybiera wszystkich następujących rodzeństwo bieżącego elementu | `//h1/following-sibling::p` | Wybiera wszystkie `<p>` będące następującymi rodzeństwem po `<h1>`. |
| `preceding-sibling::` | Wybiera wszystkich poprzednich rodzeństwo bieżącego elementu | `//h1/preceding-sibling::p` | Wybiera wszystkie `<p>` będące poprzedzającymi rodzeństwem przed `<h1>`. |
| `ancestor-or-self::` | Wybiera bieżący element oraz wszystkich jego przodków | `//span/ancestor-or-self::div` | Wybiera element `<span>` oraz wszystkie `<div>`, które są jego przodkami. |
| `descendant-or-self::` | Wybiera bieżący element oraz wszystkich jego potomków | `//div/descendant-or-self::p` | Wybiera element `<div>` oraz wszystkie `<p>`, które są jego potomkami. |
| `self::` | Wybiera tylko bieżący element | `//div/self::div` | Wybiera bieżący element `<div>` (np. jako część bardziej złożonego wyrażenia XPath). |
| `//node()` | Wybiera wszystkie węzły (elementy, tekst, komentarze, itd.) | `//node()` | Wybiera wszystkie węzły w dokumencie. |
| `comment()` | Wybiera węzły komentarzy | `//comment()` | Wybiera wszystkie komentarze w dokumencie (np. `<!-- komentarz -->`). |
| `namespace::` | Wybiera przestrzenie nazw związane z elementem | `namespace::*` | Wybiera wszystkie przestrzenie nazw z bieżącego elementu. |
| `preceding::` | Wybiera wszystkie węzły poprzedzające bieżący element | `//div/preceding::span` | Wybiera wszystkie `<span>` przed `<div>` w dokumencie. |
| `following::` | Wybiera wszystkie węzły następujące po bieżącym elemencie | `//div/following::span` | Wybiera wszystkie `<span>` po `<div>` w dokumencie. |
| `name()` | Pobiera nazwę elementu | `//input[name()='username']` | Wybiera element `<input>`, którego nazwa jest `username`. |
| `local-name()` | Pobiera nazwę lokalną elementu (bez przestrzeni nazw) | `//*[local-name()='div']` | Wybiera wszystkie elementy o lokalnej nazwie `div`, niezależnie od przestrzeni nazw. |
| `string()` | Konwertuje węzeł na ciąg tekstowy | `string(//title)` | Pobiera tekst zawarty w węźle `<title>`. |
| `normalize-space()` | Usuwa dodatkowe białe znaki z tekstu | `//p[normalize-space(.)="Hello"]` | Wybiera `<p>`, którego tekst to dokładnie `Hello` (bez dodatkowych spacji). |
| `boolean()` | Sprawdza, czy wyrażenie XPath zwraca wartość logiczną | `boolean(//input[@checked])` | Sprawdza, czy istnieje zaznaczone `<input>` z atrybutem `checked`. |
| `count()` | Liczy liczbę węzłów pasujących do wyrażenia | `count(//div)` | Zwraca liczbę wszystkich `<div>` w dokumencie. |
| `sum()` | Oblicza sumę wartości liczbowych | `sum(//price)` | Oblicza sumę wartości w węzłach `<price>`. |
| `starts-with()` | Sprawdza, czy tekst lub atrybut zaczyna się od określonego podciągu | `//input[starts-with(@id, 'user')]` | Wybiera `<input>`, którego `id` zaczyna się od `user`. |
| `substring(string, start, length)` | Wybiera podciąg tekstu od podanej pozycji na określoną długość | `substring(//title, 1, 5)` | Pobiera pierwsze 5 znaków z węzła `<title>`. |
| `substring-before()` | Pobiera część tekstu przed określonym znakiem | `substring-before(//title, '-')` | Pobiera tekst z `<title>` przed znakiem `-`. |
| `substring-after()` | Pobiera część tekstu po określonym znaku | `substring-after(//title, '-')` | Pobiera tekst z `<title>` po znaku `-`. |
| `translate()` | Zamienia określone znaki w ciągu tekstowym | `translate(//title, 'ABC', 'abc')` | Zamienia wszystkie `A`, `B`, `C` w tekście `<title>` na `a`, `b`, `c`. |
| `concat()` | Łączy ciągi tekstowe | `concat(//first-name, ' ', //last-name)` | Łączy imię i nazwisko w jeden ciąg tekstowy. |
| `boolean(value)` | Konwertuje wartość na wartość logiczną | `boolean(//input[@checked])` | Zwraca `true`, jeśli istnieje `<input>` z atrybutem `checked`. |
| `not()` | Neguje wyrażenie | `//div[not(@class='active')]` | Wybiera wszystkie `<div>`, które **nie** mają klasy `active`. |
| `div` | Dzieli jedną wartość przez drugą | `//price div 2` | Dzieli wartość w węźle `<price>` przez 2. |
| `mod` | Zwraca resztę z dzielenia | `position() mod 2 = 0` | Wybiera wszystkie elementy na parzystych pozycjach. |
| `floor()` | Zaokrągla w dół | `floor(7.8)` | Zwraca `7`. |
| `ceiling()` | Zaokrągla w górę | `ceiling(7.1)` | Zwraca `8`. |
| `round()` | Zaokrągla do najbliższej liczby całkowitej | `round(7.5)` | Zwraca `8`. |

## Z jakich metod wyszukiwania korzystać?

| Lokalizator | Zalety | Wady | Typowy przypadek użycia |
| --- | --- | --- | --- |
| Po atrybucie (id, name, class) | Łatwy w użyciu | Te atrybuty nie zawsze są dostępne | Elementy, które mają te atrybuty ustawione na stałe (nie zmieniają się dynamicznie) |
| Po tekście linku (cały lub część) | Łatwy w użyciu | Dostępny tylko dla linków | Dla linków tekstowych |
| Po nazwie tagu | Łatwy w użyciu | Trudno wybrać konkretny element, gdy tag powtarza się na stronie | Gdy tag jest unikalny lub węzeł DOM ma stałą pozycję |
| Po selektorze CSS lub XPath | Bardzo skuteczna możliwość odnajdywania | Niełatwo pisać odporne selektory | Dla złożonych lokalizatorów |
| Lokalizatory złożone (Compound locators) | Łatwy sposób na łączenie istniejących lokalizatorów | Ograniczone do specyficznych sytuacji | Gdy szukamy po id lub name (ByIdOrName), elementów zagnieżdżonych (ByChained), lub kilku strategii naraz |
| Lokalizatory relatywne (Relative locators) | Podejście zbliżone do języka naturalnego | Wymaga połączenia z innymi lokalizatorami | Znajdowanie elementu na podstawie relatywnego położenia (nad, pod, blisko, itd.) względem znanego elementu |

| Porównanie | XPath | CSS Selector |
| --- | --- | --- |
| Kierunek | Umożliwia nawigację w obie strony: od rodzica do dziecka i odwrotnie | Umożliwia nawigację tylko w jedną stronę: od rodzica do dziecka |
| Wydajność | Wolniejszy pod względem wydajności | Szybszy pod względem wydajności |

# Strategie czekania na element (Wait)

## **ImplicitWait**

`ImplicitWait` (niejawny) to mechanizm w `Selenium WebDriver`, który pozwala ustawić globalny czas oczekiwania na znalezienie elementów na stronie. Jeśli WebDriver nie znajdzie elementu od razu, będzie go szukał przez określony czas, zanim zgłosi wyjątek `NoSuchElementException`.

```java
driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
```

Dotyczy wszystkich operacji wyszukiwania elementów (`findElement`, `findElements`). Jeśli element pojawi się wcześniej, WebDriver natychmiast przejdzie dalej, bez czekania do końca ustawionego czasu. Jeśli element nie pojawi się w danym czasie, Selenium zgłosi wyjątek `NoSuchElementException`. Ustawienie `ImplicitWait` obowiązuje dla całej sesji WebDrivera, dopóki nie zostanie zmienione.

**Wady ImplicitWait  -** Może powodować nieprzewidywalne opóźnienia Jeśli element już istnieje, testy i tak czekają przez ustawiony czas. Może wystąpić także konflikt ze zdefiniowanym ExplicitWait. Gdy ImplicitWait i ExplicitWait (WebDriverWait) są używane jednocześnie, mogą prowadzić do nieoczekiwanych wyników. `ImplicitWait` działa tylko przy wyszukiwaniu elementów, ale nie pomoże np. w czekaniu na to, aż element stanie się klikalny.

## **Explicit Wait**

`ExplicitWait` (jawny) to mechanizm w Selenium WebDriver, który pozwala na czekanie na spełnienie określonych warunków przed wykonaniem kolejnych kroków w teście. Jest bardziej elastyczny niż `ImplicitWait`, ponieważ umożliwia czekanie na konkretne zdarzenia, np. aż element stanie się *klikalny* lub widoczny.

```java
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
WebElement element = wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("someElementId")));
```

`WebDriver` czeka maksymalnie określoną liczbę sekund. Jeśli warunek zostanie spełniony wcześniej, test kontynuuje bez czekania do końca. Jeśli warunek nie zostanie spełniony, Selenium zgłasza wyjątek `TimeoutException`.

✅ Używaj `ExplicitWait`, gdy elementy ładują się asynchronicznie. 

✅ Nie mieszaj `ImplicitWait` i `ExplicitWait`, bo mogą powodować nieprzewidywalne efekty.

✅ Wybieraj odpowiednie warunki z `ExpectedConditions`, by uniknąć niepotrzebnego czekania.

Wniosek: `ExplicitWait` jest bardziej elastyczny i precyzyjny niż `ImplicitWait`, dlatego w większości przypadków jest lepszym wyborem.

| **Cecha** | **`ExplicitWait`** | **`ImplicitWait`** |
| --- | --- | --- |
| Zakres | Działa na pojedynczym elemencie | Dotyczy całej sesji WebDrivera |
| Warunki | Możliwość określenia różnych warunków (`ExpectedConditions`) | Tylko czeka na znalezienie elementu |
| Wydajność | Lepsza, bo nie czeka niepotrzebnie, jeśli warunek jest spełniony wcześniej | Może powodować niepotrzebne opóźnienia |
| Obsługa dynamicznych elementów | Tak | Nie |
| Konflikty | Może kolidować z `ImplicitWait` | Może kolidować z `ExplicitWait` |

## **FluentWait**

`FluentWait` to mechanizm w `Selenium WebDriver`, który pozwala na bardziej zaawansowaną konfigurację oczekiwania na elementy. Daje większą kontrolę niż ExplicitWait, ponieważ umożliwia ustawienie interwału sprawdzania warunku oraz określenie wyjątków, które mają być ignorowane podczas czekania.

```java

FluentWait<WebDriver> wait = new FluentWait<>(driver)
        .withTimeout(Duration.ofSeconds(15))       // Maksymalny czas oczekiwania
        .pollingEvery(Duration.ofSeconds(2))       // Co ile czasu Selenium sprawdza warunek
        .ignoring(NoSuchElementException.class);

// Definicja warunku wyszukiwania elementu
WebElement element = wait.until(new Function<WebDriver, WebElement>() {
    public WebElement apply(WebDriver driver) {
        WebElement el = driver.findElement(By.id("someElementId"));
        if (el.isDisplayed()) {
            return el;
        }
        return null;
    }
});

//inny przykład
        fluentWait.until(driver -> driver.getCurrentUrl().equals("https://bonigarcia.dev/selenium-webdriver-java/login-sucess.html"));

```

| **Metoda** | **Opis** |
| --- | --- |
| `withTimeout(Duration timeout)` | Określa maksymalny czas oczekiwania na element. Czas ten jest używany do sprawdzania warunków. |
| `pollingEvery(Duration interval)` | Określa czas pomiędzy kolejnymi próbami sprawdzenia warunku. |
| `ignoring(Class<? extends Throwable>... ignoredExceptions)` | Określa wyjątki, które powinny być ignorowane podczas oczekiwania (np. `NoSuchElementException`). |
| `ignoreAll(Iterable<Class<? extends Throwable>> ignoredExceptions)` | Określa zestaw wyjątków, które mają być ignorowane podczas oczekiwania. |
| `until(Condition<? super T> condition)` | Określa warunek, który ma być spełniony, aby zakończyć oczekiwanie. Warunek jest regularnie sprawdzany. |

Jak działa FluentWait? Czeka maksymalnie `withTimeout()` czasu (np. 15 sekund). Co określony czas `pollingEvery()` sprawdza warunek (np. co 2 sekundy). Ignoruje podane wyjątki `ignoring()`, np. `NoSuchElementException`. Gdy warunek zostanie spełniony, kończy czekanie i zwraca element.

Zalety FluentWait: Pozwala dynamicznie sprawdzać warunek zamiast czekać bezczynnie.  Można ustawić częstotliwość sprawdzania (nie musi czekać do końca czasu jak ExplicitWait). Można ignorować określone wyjątki, unikając niepotrzebnych błędów.

FluentWait daje największą elastyczność – pozwala ignorować wyjątki i ustawiać interwały sprawdzania.  Jest szczególnie przydatny dla dynamicznych stron, gdzie elementy mogą pojawiać się w różnych momentach.  Jeśli nie potrzebujesz takiej kontroli, `ExplicitWait` będzie prostszym wyborem.

Najlepsza praktyka: używać FluentWait w przypadkach, gdy elementy ładują się nieregularnie i chcesz uniknąć ciągłego sprawdzania.

| **Cecha** | **`FluentWait`** | **`ExplicitWait`** | **`ImplicitWait`** |
| --- | --- | --- | --- |
| Zakres | Pojedynczy element | Pojedynczy element | Wszystkie operacje wyszukiwania |
| Warunki | Można definiować własne | `ExpectedConditions` | Brak warunków – tylko obecność elementu |
| Częstotliwość sprawdzania | Można ustawić (`pollingEvery()`) | Sprawdza co określony czas | N/A |
| Obsługa wyjątków | Można ignorować określone wyjątki | Nie ignoruje wyjątków domyślnie | Nie ignoruje wyjątków |
| Wydajność | Bardziej efektywny dla dynamicznych stron | Może czekać do końca maksymalnego czasu | Może powodować opóźnienia |
| Obsługa dynamicznych elementów | Tak | Tak | Nie |

# JavaScript

Możemy używać JavaScript po stronie klienta aplikacji internetowych do wielu różnych operacji, takich jak manipulacja DOM, interakcja z użytkownikiem, obsługa żądań i odpowiedzi z zdalnych serwerów czy praca z wyrażeniami regularnymi, a także wielu innych funkcji. Na szczęście dla automatyzacji testów, Selenium WebDriver pozwala na wstrzykiwanie i wykonywanie dowolnych fragmentów JavaScript. W tym celu API Selenium WebDriver udostępnia interfejs **`JavascriptExecutor`**.

Aby uruchomić JavaScript Executor w Selenium, należy użyć interfejsu `JavascriptExecutor`. Wykorzystuje się go do wykonywania skryptów JavaScript w kontekście przeglądarki sterowanej przez `Selenium WebDriver`.

```java
JavascriptExecutor js = (JavascriptExecutor) driver;
// Przewinięcie strony w dół o 500 pikseli
js.executeScript("window.scrollBy(0,500);");

// Pobranie tytułu strony
String title = (String) js.executeScript("return document.title;");
System.out.println("Tytuł strony: " + title);
```

| **Typ skryptu** | **Metoda w Selenium** | **Zachowanie** |
| --- | --- | --- |
| **Synchroniczny** | `executeScript(String script, Object... args)` | Skrypt JS wykonuje się **od razu**, a Selenium czeka na jego zakończenie przed wykonaniem kolejnej akcji. |
| **Asynchroniczny** | `executeAsyncScript(String script, Object... args)` | Skrypt może trwać **dłużej** (np. czekać na operacje sieciowe), a Selenium czeka na ręczne wywołanie `arguments[arguments.length - 1]` jako callback. |

## **Synchroniczny skrypt w `JavascriptExecutor`**

**Przyjmowane argumenty (`executeScript`)**

Metoda `executeScript(String script, Object... args)` może przyjmować różne typy argumentów, które są przekazywane do skryptu JavaScript jako `arguments[i]`:

1. **Liczby (`Number`)** – przekazywane jako `Double`.
2. **Ciągi znaków (`String`)** – przekazywane bez zmian.
3. **Wartości logiczne (`Boolean`)** – przekazywane bez zmian.
4. **Elementy `WebElement`** – zamieniane na obiekty `HTMLElement` w JavaScript.
5. **Tablice (`List<Object>`)** – konwertowane na tablice JavaScript.
6. **Mapy (`Map<String, Object>`)** – konwertowane na obiekty JSON.

**Zwracany typ:**

- `executeScript` może zwracać różne wartości w zależności od działania skryptu:
- **`Long`** – jeśli skrypt zwróci liczbę całkowitą.
- **`Double`** – jeśli skrypt zwróci liczbę zmiennoprzecinkową.
- **`Boolean`** – jeśli skrypt zwróci wartość logiczną.
- **`String`** – jeśli skrypt zwróci tekst.
- **`List<Object>`** – jeśli skrypt zwróci tablicę JavaScript.
- **`Map<String, Object>`** – jeśli skrypt zwróci obiekt JSON.
- **`WebElement`** – jeśli skrypt zwróci element DOM.
- **`null`** – jeśli skrypt nie zwróci wartości.

## **Argumenty w JS**

```java
String script = "arguments[0].style.backgroundColor = arguments[1];";
js.executeScript(script, element, "yellow");
```

# Timeouts

Określa maksymalny czas oczekiwania na pełne załadowanie strony przed wykonaniem dalszych operacji. Limit czasu ładowania strony zapewnia limit czasu na przerwanie próby nawigacji. Innymi słowy, limit czasu ogranicza czas ładowania strony internetowej. Po przekroczeniu tego limitu czasu (którego domyślna wartość wynosi 30 sekund) zgłaszany jest wyjątek. 

```java
driver.manage().timeouts().pageLoadTimeout(Duration.ofSeconds(30));

```

Zalety: Zapobiega długiemu oczekiwaniu na wolno ładujące się strony Wady: Może przerwać ładowanie strony, jeśli ustawiony czas jest za krótki

**Limit czasu ładowania skryptu zapewnia limit czasu na przerwanie ocenianego skryptu. Ten limit czasu ma domyślną wartość trzystu sekund.** 

```java
JavascriptExecutor js = (JavascriptExecutor) driver;
driver.manage().timeouts().scriptTimeout(Duration.ofSeconds(3));
```

# Shadow DOM

Shadow DOM to mechanizm enkapsulacji w HTML i JavaScript, który pozwala na tworzenie komponentów o własnym drzewie DOM, izolowanych od reszty strony. Dzięki temu elementy wewnątrz Shadow DOM: Nie wpływają na globalne style strony. Nie są dostępne za pomocą standardowego `document.querySelector()`. Są niezależne od innych skryptów i CSS.

Shadow DOM to mechanizm Web Components, który pozwala na enkapsulację struktury HTML i stylów w elemencie. Elementy wewnątrz shadow root nie są bezpośrednio dostępne za pomocą standardowych metod findElement().

| **Metoda** | **Opis** |
| --- | --- |
| `driver.findElement(By.cssSelector())` | Nie działa na elementy wewnątrz Shadow DOM. |
| `shadowHost.getShadowRoot()` | Pobiera root Shadow DOM dla elementu będącego hostem. |
| `shadowRoot.findElement(By.cssSelector())` | Wyszukuje elementy wewnątrz Shadow DOM. |
| `driver.executeScript("return arguments[0].shadowRoot", element)` | Pobiera shadow root za pomocą JavaScript. |

Jeśli strona zawiera elementy wewnątrz shadow root, Selenium nie znajdzie ich w standardowy sposób:

```html
<div id="shadow-host">
    #shadow-root (open)
    <input id="shadow-input" type="text">
</div>
```

```java
WebElement input = driver.findElement(By.id("shadow-input")); // Nie zadziała!
```

Poprawne użycie:

```java
WebElement shadowHost = driver.findElement(By.id("shadow-host"));
WebElement shadowRoot = (WebElement) ((JavascriptExecutor) driver)
        .executeScript("return arguments[0].shadowRoot", shadowHost);
WebElement input = shadowRoot.findElement(By.id("shadow-input"));
input.sendKeys("Test Selenium");
```

W Selenium 4+ dodane zostały dodatkowe narzędzia pozwalające na przeszukiwanie shadowDOM wbudowanymi metodami. Przykład użycia: 

```java
WebElement shadowHost  = driver.findElement(By.id("content"));
SearchContext shadowRoot = shadowHost.getShadowRoot();
WebElement paragraph = shadowRoot.findElement(By.cssSelector("p"));
```

💡Uwaga! W Selenium 4 + ChromeDriver, **wyszukiwanie poprzez XPath w `shadowRoot` często powoduje błąd `InvalidArgumentException`  dlatego wyszukiwanie nie zadziała**, **bo XPath w shadowRoot jest niewspierany**.

# **Ciasteczka (cookies)**

HTTP 1.x to protokół bezstanowy, co oznacza, że serwer nie śledzi stanu użytkownika. Innymi słowy, serwery internetowe nie zapamiętują użytkowników w różnych żądaniach. Mechanizm plików cookie jest rozszerzeniem protokołu HTTP, które umożliwia śledzenie użytkowników poprzez wysyłanie małych fragmentów tekstu zwanych plikami cookie z serwera do klienta. Te pliki cookie muszą być wysyłane z powrotem przez klientów, dzięki czemu serwery zapamiętują swoich klientów. Pliki cookie umożliwiają między innymi utrzymywanie sesji internetowych lub personalizowanie doświadczenia użytkownika na stronie internetowej. Przeglądarki internetowe umożliwiają ręczne zarządzanie plikami cookie przeglądarki. Selenium WebDriver umożliwia równoważną manipulację, ale programowo. Metody dotyczące ciasteczek są dostępne poprzez funkcję `manage()` obiektu WebDriver.

Z ciasteczkami pracujemy za pomocą `driver.manage()` a także klasy `Cookie` w Java. 

Aby zmienić wartości ciasteczka, stare musi zostać usunięte i utworzone nowe. 

**Metody zarządzania cookies w Selenium**

| **Metoda** | **Opis** |
| --- | --- |
| `getCookies()` | Pobiera wszystkie ciasteczka dostępne w danej sesji przeglądarki. Zwraca `Set<Cookie>`. |
| `getCookieNamed(String name)` | Pobiera konkretne ciasteczko na podstawie jego nazwy. Zwraca obiekt `Cookie` lub `null`, jeśli ciasteczko nie istnieje. |
| `addCookie(Cookie cookie)` | Dodaje nowe ciasteczko do sesji przeglądarki. Wymaga obiektu `Cookie`. |
| `deleteCookie(Cookie cookie)` | Usuwa określone ciasteczko, podając jego obiekt `Cookie`. |
| `deleteCookieNamed(String name)` | Usuwa ciasteczko o podanej nazwie. |
| `deleteAllCookies()` | Usuwa wszystkie ciasteczka dostępne w danej sesji przeglądarki. |

**📌 Elementy ciasteczka (`Cookie`) w Selenium**

| **Element** | **Typ danych** | **Opis** | **Wymagane?** |
| --- | --- | --- | --- |
| `name` | `String` | Nazwa ciasteczka (np. `"session_id"`). | ✅ Tak |
| `value` | `String` | Wartość przechowywana w ciasteczku (np. `"abc123xyz"`). | ✅ Tak |
| `domain` | `String` | Domena, dla której ciasteczko jest ważne (np. `"example.com"`). | ❌ Nie (domyślnie aktualna domena) |
| `path` | `String` | Ścieżka, w której ciasteczko jest dostępne (np. `"/user"`). | ❌ Nie (domyślnie `"/"`) |

# Ramki (iFrame)

Podczas nawigacji po stronach internetowych za pomocą przeglądarki domyślnie używamy jednej strony odpowiadającej adresowi URL na pasku nawigacyjnym. Następnie możemy otworzyć kolejną stronę w nowej karcie przeglądarki. Ta druga karta może zostać otwarta jawnie, gdy link definiuje atrybut docelowy, lub użytkownik może wymusić nawigację do nowej karty, zwykle za pomocą klawisza modyfikatora Ctrl (lub Cmd w macOS) wraz z kliknięciem myszą w link internetowy. Inną możliwością jest otwieranie stron internetowych w nowych oknach. W tym celu strony internetowe zazwyczaj używają polecenia JavaScript `window.open(url)`. Innym sposobem wyświetlania różnych stron w tym samym czasie jest użycie ramek i ramek iFrame. Ramka jest typem elementu HTML, który definiuje określony obszar (w zestawie zwanym frameset), w którym wyświetlana jest strona internetowa.

Z ramkami pracujemy poprzez `driver.swiitchTo()`. W przypadku pracy na wielu ramkach, należy przy wychodzeniu z jednej z nich przenieść się do ramki nadrzędnej za pomocą `defaultContent().`

# Web Storage

Web Storage to mechanizm przechowywania danych w przeglądarce, który umożliwia zapisywanie danych po stronie klienta. Składa się z dwóch typów pamięci:

`localStorage` – dane są przechowywane na stałe, nawet po zamknięciu przeglądarki. 

`sessionStorage` – dane są przechowywane tylko na czas trwania sesji, czyli do zamknięcia karty lub przeglądarki. 

Jest szybszy niż cookies i może przechowywać większą ilość danych (zwykle do 5MB).

| **Komenda** | **Opis** |
| --- | --- |
| `localStorage.getItem('klucz')` | Pobiera wartość z Local Storage |
| `localStorage.setItem('klucz', 'wartość')` | Zapisuje wartość w Local Storage |
| `localStorage.removeItem('klucz')` | Usuwa wpis z Local Storage |
| `localStorage.clear()` | Czyści cały Local Storage |
| `sessionStorage.getItem('klucz')` | Pobiera wartość z Session Storage |
| `sessionStorage.setItem('klucz', 'wartość')` | Zapisuje wartość w Session Storage |
| `sessionStorage.removeItem('klucz')` | Usuwa wpis z Session Storage |
| `sessionStorage.clear()` | Czyści cały Session Storage |

Tradycyjne klasy Selenium związane z zarządzaniem pamięcią przeglądarki zostały oznaczone jako zgłoszone do usunięcia https://www.selenium.dev/selenium/docs/api/java/org/openqa/selenium/html5/Storage.html. Rekomendowane jest w dokumentacji Selenium korzystanie z JavaScript.

# Event Listeners

API Selenium WebDriver pozwala na tworzenie nasłuchiwaczy, które powiadamiają o zdarzeniach zachodzących w WebDriverze i obiektach pochodnych. Event Listeners w Selenium to mechanizm, który pozwala przechwytywać i reagować na różne zdarzenia związane z WebDriverem, np. kliknięcia, zmiany adresu URL czy błędy. W poprzednich wersjach Selenium WebDriver funkcja ta była dostępna za pośrednictwem klasy EventFiringWebDriver. Klasa ta jest przestarzała od wersji Selenium WebDriver 4, a zamiast niej powinniśmy używać następującej: 

- `EventFiringDecorator` - Wrapper class dla WebDrivera i obiektów pochodnych (np. WebElement, TargetLoca tor, itp.). Umożliwia ona rejestrację jednego lub więcej słuchaczy (tj. instancji WebDriverListener).
- `WebDriverListener` -  Interfejs, który powinien implementować słuchaczy zarejestrowanych w dekoratorze. Obsługuje trzy typy zdarzeń: **Before events** Logika wstawiana tuż przed rozpoczęciem jakiegoś zdarzenia **After events** Logika wstawiana tuż po zakończeniu jakiegoś zdarzenia **Error events** Logika wstawiana przed rzuceniem wyjątku Aby zaimplementować słuchacza zdarzeń, najpierw powinniśmy utworzyć klasę słuchacza. Innymi słowy, musimy stworzyć klasę, która implementuje interfejs WebDriverListener. Interfejs ten definiuje wszystkie swoje metody za pomocą domyślnego słowa kluczowego, a zatem nadpisywanie ich metod jest opcjonalne.

Aby zaimplementować słuchacza zdarzeń, najpierw powinniśmy utworzyć klasę słuchacza. Innymi słowy, musimy utworzyć klasę, która implementuje interfejs WebDriverListener.

```java
public class Listener implements WebDriverListener {

@Override
    public void onError(Object target, Method method, Object[] args, InvocationTargetException e) {
        log.error("Wystąpił błąd w teście, dotyczący {}, użyto metody {}. Komunikat z błędem: {}", target, method.getName(), e.toString());
    }
   ....
```

 Interfejs ten definiuje wszystkie swoje metody za pomocą domyślnego słowa kluczowego, a zatem nadpisywanie ich metod jest opcjonalne. Dzięki tej funkcji (dostępnej od Javy 8), nasza klasa powinna implementować tylko te metody, których potrzebujemy. Dostępnych jest wiele metod nasłuchujących, na przykład `afterGet()` (wykonywana po wywołaniu metody get() w instancji WebDriver) lub `beforeQuit()` (wykonywana przed wywołaniem metody `quit()` w instancji WebDriver), aby wymienić tylko kilka. 

```java
@Override
public void beforeClickOn(WebElement element, WebDriver driver) {
    System.out.println("Przed kliknięciem: " + element.getText());
}

@Override
public void afterClickOn(WebElement element, WebDriver driver) {
    System.out.println("Po kliknięciu: " + element.getText());
}
```

# WebDriver Exceptions

- Wszystkie wyjątki dostarczane przez API WebDriver dziedziczą z klasy WebDriver Exception i są odznaczone (*unchecked*).
    
    Wyjątki w Javie to zdarzenia, które przerywają normalny przebieg programu podczas jego wykonywania. Hierarchia klas wyjątków w Javie opiera się na klasie bazowej Throwable, która dzieli się na dwie główne podklasy: Error i Exception. Error Reprezentuje poważne, często nieodwracalne problemy systemowe, których aplikacja nie powinna obsługiwać. Przykłady to OutOfMemoryError czy StackOverflowError. Zazwyczaj prowadzą one do przerwania działania programu i nie są przechwytywane w kodzie aplikacji. Exception Oznacza sytuacje, które mogą być obsłużone przez aplikację. Wyjątki te dzielą się na dwa typy: • Wyjątki sprawdzane (checked exceptions) Są to klasy dziedziczące po Exception, ale nie po RuntimeException. Kompilator wymaga, aby takie wyjątki były obsłużone przez blok try-catch lub zadeklarowane w sygnaturze metody za pomocą słowa kluczowego throws. Przykłady: IOException, MalformedURLException, ClassNotFoundException. • Wyjątki niesprawdzane (unchecked exceptions) Są to klasy dziedziczące po RuntimeException. Kompilator nie wymusza ich obsługi ani deklarowania. Zazwyczaj wynikają z błędów programistycznych, takich jak NullPointerException czy ArrayIndexOutOfBoundsException.
    

| Wyjątek | Opis (PL) | Typowe przyczyny (PL) |
| --- | --- | --- |
| `NoSuchElementException` | Element nie jest dostępny | - Nieprawidłowa strategia lokalizacji<br>- Element nie został wyrenderowany (może trzeba poczekać) |
| `NoAlertPresentException` | Okno dialogowe (alert, prompt lub confirm) nie jest dostępne | - Próba wykonania akcji (np. accept() lub dismiss()) na nieistniejącym oknie dialogowym |
| `NoSuchWindowException` | Okno lub karta nie jest dostępna | - Próba przełączenia się do nieistniejącego okna lub karty |
| `NoSuchFrameException` | Ramka (frame lub iframe) nie jest dostępna | - Próba przełączenia się do nieistniejącej ramki lub iframe |
| `InvalidArgumentException` | Nieprawidłowy argument podczas wywołania metody Selenium WebDriver | - Błędny URL w metodach nawigacyjnych<br>- Nieistniejąca ścieżka podczas uploadu pliku<br>- Zły typ argumentu w skrypcie JavaScript |
| `StaleElementReferenceException` | Element jest „przestarzały” – nie występuje już na stronie | - Zmiana DOM podczas próby interakcji z wcześniej znalezionym elementem |

| Ograniczenie wydajności CPU | Spowolnienie działania procesora |
| Symulacja geolokalizacji | Ustawienie własnych współrzędnych GPS |
| Ustawienie orientacji | Obracanie ekranu (np. pion/poziom) |

**Automatyzacja emulacji urządzeń mobilnych przez Selenium WebDriver:**

- W przeglądarkach opartych na Chromium (np. Chrome, Edge) można automatyzować emulację urządzeń mobilnych za pomocą Selenium WebDriver, ustawiając opcję „mobileEmulation” w ChromeOptions lub EdgeOptions.
- Są dwie metody konfiguracji:
    - Wskazanie konkretnego urządzenia (np. Pixel 2, iPad Pro, Galaxy Fold) – lista dostępnych urządzeń jest aktualizowana w każdej wersji Chromium.
    - Wskazanie indywidualnych parametrów urządzenia: szerokość (width), wysokość (height), współczynnik pikseli (pixelRatio), obsługa dotyku (touch), a także user agent.

**Czy emulacja działa w Firefoxie?**

- Emulacja urządzeń mobilnych w pełnym zakresie (tak jak w Chrome, z obsługą mobileEmulation przez API Selenium WebDriver) NIE jest dostępna w Firefoxie, ponieważ nie została zaimplementowana w geckodriver (sterowniku Selenium dla Firefoxa).
- W Firefoxie można jedynie ustawić rozmiar okna (viewport), co częściowo imituje widok mobilny, ale nie zapewnia pełnej emulacji urządzenia (np. dotyku, user agenta, pixel ratio).
- Potwierdzają to oficjalne dokumentacje i praktyka testerska – pełna automatyczna emulacja urządzeń mobilnych przez Selenium WebDriver jest możliwa tylko w przeglądarkach opartych na Chromium (Chrome, Edge), a nie w Firefoxie.

```java
/////
```

## Notyfikacje

Inaczej powiadomienia, jest to standardowy interfejs API sieci Web, który umożliwia witrynom wysyłanie powiadomień wyświetlanych na pulpicie systemu operacyjnego. Ten interfejs API jest dostępny za pośrednictwem obiektu JavaScript Notification. Zanim strona internetowa będzie mogła wysyłać powiadomienia, użytkownik musi udzielić przeglądarce pozwolenia. API Selenium WebDriver pozwala na przyznawanie uprawnień za pomocą opcji. Podobnie jak w przypadku innych funkcji, składnia tych możliwości różni się w Chrome/Edge i Firefox. W Chrome  i Edge, które są oparte na Chromium, używamy tej samej opcji ustawień (`profile.default_content_setting_val ues.notifications`), aby zezwolić na powiadomienia. W Firefox etykieta preferencji (`permissions.default.desktopnotification`) jest w tym przypadku inna, chociaż jej wartość (1) jest taka sama dla zezwalania na powiadomienia. Inną możliwą wartością jest 2, która służy do blokowania powiadomień (zarówno w Chrome/Edge, jak i Firefox).

```java
ChromeOptions options = new ChromeOptions();
Map<String, Object> prefs = new HashMap<>();
prefs.put("profile.default_content_setting_values.notifications", 1);
options.setExperimentalOption("prefs", prefs);
driver = WebDriverManager.chromedriver().capabilities(options).create();
```

```java
FirefoxOptions options = new FirefoxOptions();
options.addPreference("permissions.default.desktop-notification", 1);
driver = WebDriverManager.firefoxdriver().capabilities(options).create();
```

## Lokalizacja (język)

W inżynierii oprogramowania lokalizacja odnosi się do procesu dostosowywania aplikacji do kultury i języka (zwanego `locale`) użytkowników końcowych. Lokalizacja jest czasami zapisywana jako `l10n` (10 to liczba liter między l i n w angielskim słowie localization). Najbardziej typowym działaniem lokalizacyjnym jest tłumaczenie tekstu wyświetlanego w interfejsie użytkownika aplikacji na różne języki. Ponadto inne aspekty interfejsu użytkownika można dostosować w zależności od lokalizacji, takie jak waluty (euro, dolary itp.), systemy miar (np. systemy metryczne lub imperialne) lub format liczb i dat. L10n jest częścią szerszej koncepcji zwanej internacjonalizacją (i18n), która jest procesem projektowania i rozwijania aplikacji, która umożliwia łatwe l10n dla heterogenicznych odbiorców docelowych. Powszechne praktyki umożliwiające `i18n` to używanie Unicode do kodowania tekstu lub dodawanie obsługi CSS dla tekstu pionowego lub typografii innych niż łacińskie. Testowanie lokalizacji jest formą testowania niefunkcjonalnego, w którym SUT jest weryfikowany pod kątem określonych ustawień regionalnych. API Selenium WebDriver pozwala nam na testowanie lokalizacji w oparciu o język przeglądarki poprzez ustawienie właściwości intl.accept_languages. Właściwość ta pozwala na określenie identyfikatora ustawień regionalnych, takich jak en_US dla amerykańskiego angielskiego lub es_ES dla europejskiego hiszpańskiego.

```java
@BeforeEach
void setup() {
lang = "es-ES";
ChromeOptions options = new ChromeOptions();
Map<String, Object> prefs = new HashMap<>();
prefs.put("intl.accept_languages", lang);
options.setExperimentalOption("prefs", prefs);
driver = WebDriverManager.chromedriver().capabilities(options).create();
```

```java
@BeforeEach
void setup() {
lang = "es-ES";
FirefoxOptions options = new FirefoxOptions();
options.addPreference("intl.accept_languages", lang);
driver = WebDriverManager.firefoxdriver().capabilities(options)
.create();
}
```

Istnieje druga alternatywa dla testowania lokalizacji za pomocą Selenium Web Driver. Zamiast zmieniać preferowany język (który określa nagłówek HTTP accept-language), możemy zmienić domyślny język przeglądarki internetowej.

```java
@BeforeEach
void setup() {
lang = "en-US";
ChromeOptions options = new ChromeOptions();
options.addArguments("--lang=" + lang);
driver = WebDriverManager.chromedriver().capabilities(options).create();
}
```

# **Chrome DevTools**

Chrome DevTools to zestaw narzędzi deweloperskich wbudowanych w przeglądarki oparte na Chromium, takie jak Chrome i Edge. Umożliwiają one m.in. inspekcję, debugowanie i profilowanie stron internetowych oraz samej przeglądarki.

**Chrome DevTools Protocol (CDP)** to protokół komunikacyjny, który pozwala zewnętrznym klientom na manipulowanie przeglądarką poprzez DevTools. Dzięki CDP możliwa jest automatyzacja zadań, takich jak:

- Inspekcja i kontrola dokumentów
- Instrumentacja przeglądarki
- Symulacja interakcji użytkownika
- Nasłuchiwanie i modyfikacja zdarzeń sieciowych i logów konsoli

**CDP w Selenium WebDriver**

Od wersji 4 Selenium WebDriver umożliwia korzystanie z CDP na dwa sposoby:

**1. Interfejs HasDevTools**

- Udostępniony w sterownikach ChromiumDriver (dla Chrome i Edge) oraz FirefoxDriver (dla Firefoxa, który implementuje tylko wybrany podzbiór CDP).
- Pozwala na bezpośrednie wysyłanie komend CDP do przeglądarki przez obiekt DevTools.
- Jest to rozwiązanie bardzo elastyczne, ale zależne od typu i wersji przeglądarki[5](https://blog.qalabs.pl/junit-selenium/selenium-4-chrome-devtools-protocol/).

**2. Wrappery CDP w Selenium**

- Selenium udostępnia zestaw klas-wrappersów, które upraszczają korzystanie z wybranych funkcji CDP.
- Pozwalają na zaawansowaną manipulację przeglądarką, np. przechwytywanie ruchu sieciowego czy obsługę autoryzacji (Basic/Digest).
- Rozwiązanie to jest wygodniejsze i bardziej odporne na zmiany wersji przeglądarki, ale mniej elastyczne niż bezpośrednie wysyłanie komend CDP.

Baza:

```java
    DevTools devTools = ((ChromeDriver) driver).getDevTools();
    devTools.createSession(); // Nawiązuje połączenie z przeglądarką dla bieżącej lub określonej zakładki .
		devTools.send(Command<X> command) //Wysyła konkretne polecenie do przeglądarki i czeka na odpowiedź 
		devTools.addListener(Event<X> event, ...) //Rejestruje "nasłuchiwacz" na konkretne zdarzenia wysyłane przez przeglądarkę (np. gdy zostanie wysłane żądanie sieciowe) 
		getDomains()// Zwraca obiekt, który daje dostęp do wszystkich domen w bardziej uporządkowany sposób .
```

**Uwierzytelnianie podstawowe i skrócone**

 Protokół HTTP udostępnia dwa wbudowane mechanizmy rozpoznawania tożsamości użytkownika, zwane uwierzytelnianiem podstawowym i skróconym. Obie metody pozwalają na określenie poświadczeń użytkownika za pomocą pary wartości: nazwy użytkownika i hasła. Różnica między nimi polega na sposobie przekazywania poświadczeń. Z jednej strony, metoda uwierzytelniania skrótu wysyła zaszyfrowane poświadczenia poprzez zastosowanie funkcji skrótu do nazwy użytkownika i hasła. Z drugiej strony, podstawowe uwierzytelnianie wykorzystuje Base64 do kodowania (nie szyfrowania) poświadczeń.

```java
@Test
void testBasicAuth() {
((HasAuthentication) driver)
		.register(() -> new UsernameAndPassword("guest", "guest"));
driver.get("https://jigsaw.w3.org/HTTP/Basic/");
WebElement body = driver.findElement(By.tagName("body"));
assertThat(body.getText()).contains("Your browser made it!");
}
```

W przypadku korzystania z innych przeglądarek (takich jak Firefox), nie możemy rzutować obiektu sterownika na HasAuthentication. Niemniej jednak istnieje ogólny sposób wysyłania danych uwierzytelniających w adresie URL przy użyciu składni protocol://username:password@domain.

```java
@Test
void testGenericAuth() {
```

CDP Raw Commands Począwszy od wersji 4, Selenium WebDriver zapewnia interfejs HasDevTools do bezpośredniego korzystania z CDP. Interfejs ten jest implementowany przez ChromiumDriver (używany dla Chrome i Edge) i FirefoxDriver (dla Firefox). Aby skorzystać z tej funkcji, musimy najpierw otworzyć sesję CDP (tj. połączenie WebSocket między klientem a przeglądarką) za pomocą metody createSession() instancji DevTools. Sesja CDP jest tworzona w konfiguracji testu i zamykana w zakończeniu. Każdy test będzie używał atrybutu klasy devTools do interakcji z Chrome DevTools.

```java
		}
```

CDP (Chrome DevTools Protocol) umożliwia emulację różnych sieci (takich jak 2G/3G/4G, WiFi, Bluetooth i inne) oraz warunków połączenia (np. opóźnienia czy przepustowości). Funkcja ta jest przydatna do testowania zachowania aplikacji webowych w określonych warunkach sieciowych.

**Network.enable()**

Aktywuje śledzenie sieci. Polecenie to przyjmuje trzy opcjonalne argumenty:

- **`Optional<Integer> maxTotalBufferSize`** – maksymalny rozmiar bufora (w bajtach) dla wszystkich danych sieciowych.
- **`Optional<Integer> maxResourceBufferSize`** – maksymalny rozmiar bufora (w bajtach) dla pojedynczego zasobu.
- **`Optional<Integer> maxPostDataSize`** – maksymalny rozmiar ciała żądania POST (w bajtach).

**Network.emulateNetworkConditions()**

Aktywuje emulację warunków sieciowych. Parametry emulacji to:

- **`Boolean offline`** – emulacja braku połączenia z Internetem.
- **`Number latency`** – minimalne opóźnienie (w ms) od żądania do odpowiedzi.
- **`Number downloadThroughput`** – maksymalna przepustowość pobierania (w bajtach/sekundę); wartość -1 wyłącza ograniczenie.
- **`Number uploadThroughput`** – maksymalna przepustowość wysyłania (w bajtach/sekundę); wartość -1 wyłącza ograniczenie.
- **`Optional<ConnectionType> connectionType`** – emulowana technologia połączenia. Wyliczenie **`ConnectionType`** przyjmuje następujące opcje: NONE, CELLULAR2G, CELLULAR3G, CELLULAR4G, BLUETOOTH, ETHERNET, WIFI, WIMAX oraz OTHER.

Dzięki temu można precyzyjnie testować aplikacje webowe, symulując różne scenariusze łączności, np. wolne połączenie 2G, brak internetu lub ograniczoną przepustowość WiFi, co pozwala na lepsze przygotowanie aplikacji na realne warunki użytkowania

## Web Authentication

Web Authentication API (WebAuthn) to specyfikacja W3C, która umożliwia serwerom rejestrowanie i uwierzytelnianie użytkowników za pomocą kryptografii klucza publicznego zamiast haseł. Dzięki temu rozwiązaniu użytkownicy mogą logować się do serwisów internetowych bez konieczności podawania hasła, wykorzystując do tego celu urządzenia uwierzytelniające, takie jak klucze bezpieczeństwa USB, NFC, czy wbudowane czytniki biometryczne (np. Touch ID, Windows Hello).

**Jak działa WebAuthn?**

- Podczas rejestracji serwer za pomocą Web Authentication API prosi użytkownika o utworzenie pary kluczy (prywatnego i publicznego), tzw. credential.
- Klucz prywatny jest bezpiecznie przechowywany na urządzeniu użytkownika (np. w kluczu sprzętowym lub w systemie operacyjnym), a klucz publiczny trafia na serwer.
- Podczas logowania serwer wysyła wyzwanie (challenge), które użytkownik podpisuje swoim kluczem prywatnym. Serwer weryfikuje podpis przy użyciu klucza publicznego i na tej podstawie potwierdza tożsamość użytkownika[10](https://webauthn.guide/)[12](https://en.wikipedia.org/wiki/WebAuthn)[17](https://developer.mozilla.org/en-US/docs/Web/API/Web_Authentication_API).

**Zalety WebAuthn:**

- Zwiększone bezpieczeństwo: brak przesyłania haseł, odporność na phishing i ataki typu man-in-the-middle.
- Wygoda: użytkownik nie musi pamiętać haseł, może używać np. odcisku palca lub klucza sprzętowego.
- Uniwersalność: obsługiwany przez wszystkie główne przeglądarki (Chrome, Firefox, Edge, Safari) oraz systemy mobilne[6](https://www.nevis.net/en/fundamentals/what-is-webauthn)[8](https://auth0.com/blog/using-webauthn-to-secure-your-digital-life/).

Od wersji 4 Selenium WebDriver wspiera WebAuthn dzięki interfejsowi HasVirtualAuthenticator. Pozwala to na automatyzację testów aplikacji korzystających z WebAuthn bez konieczności fizycznego używania kluczy sprzętowych – Selenium umożliwia tworzenie wirtualnych authenticatorów, które symulują zachowanie prawdziwych urządzeń.

- Funkcja Virtual Authenticator jest dostępna tylko w przeglądarkach opartych na Chromium (Chrome, Edge)[1](https://www.selenium.dev/documentation/webdriver/interactions/virtual_authenticator/)[4](https://github.com/SeleniumHQ/selenium/issues/10541).
- Można skonfigurować rodzaj protokołu (np. CTAP2), typ transportu (USB, NFC, internal), obecność klucza rezydentnego, czy użytkownik jest zweryfikowany itp.
- Przykładowe użycie w JavaScript polega na dodaniu wirtualnego authenticatora przed wywołaniem kodu korzystającego z **`navigator.credentials.create`** lub **`navigator.credentials.get`**

```java
@Test
void testWebAuthn() {
driver.get("https://webauthn.io/");
HasVirtualAuthenticator virtualAuth = (HasVirtualAuthenticator) driver;
VirtualAuthenticator authenticator = virtualAuth
.addVirtualAuthenticator(new VirtualAuthenticatorOptions());
String randomId = UUID.randomUUID().toString();
driver.findElement(By.id("input-email")).sendKeys(randomId);
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(20));
driver.findElement(By.id("register-button")).click();
wait.until(ExpectedConditions.textToBePresentInElementLocated(
By.className("popover-body"), "Success! Now try logging in"));
driver.findElement(By.id("login-button")).click();
wait.until(ExpectedConditions.textToBePresentInElementLocated(
By.className("main-content"), "You're logged in!"));
virtualAuth.removeVirtualAuthenticator(authenticator);
}
```

# WebDriver BiDi

WebDriver BiDi to szkic specyfikacji W3C, który definiuje dwukierunkowy protokół WebDriver. Zamiast ścisłego formatu polecenie/odpowiedź protokołu WebDriver, BiDi wprowadza połączenie WebSocket między sterownikiem a przeglądarką, aby umożliwić dwukierunkową komunikację. W ten sposób WebDriver BiDi pozwoli na wykonywanie różnych operacji przy użyciu szybkiego, dwukierunkowego transportu (czyli bez odpytywania przeglądarki w celu uzyskania odpowiedzi). W Selenium WebDriver celem jest, aby BiDi w dłuższej perspektywie stał się standardowym zamiennikiem zaawansowanych operacji obecnie obsługiwanych przez CDP. Na przykład API Selenium WebDriver umożliwia implementację nasłuchiwaczy zdarzeń poprzez interfejs HasLogEvents. Interfejs ten działa obecnie w oparciu o CDP. Niemniej jednak w przyszłych wersjach Selenium WebDriver będzie wykorzystywał BiDi wewnętrznie, zapewniając bardziej niezawodną kompatybilność między przeglądarkami. HasLogEvents pozwala na implementację nasłuchiwaczy dla następujących zdarzeń:

[BiDi](Selenium%20JS/BiDi%202c83f0d8989e80538c2ee33b536dfd34.md)

```java
@Test
void testDomMutation() throws InterruptedException {
driver.get("https://bonigarcia.dev/selenium-webdriver-java/");
HasLogEvents logger = (HasLogEvents) driver;
JavascriptExecutor js = (JavascriptExecutor) driver;
AtomicReference<DomMutationEvent> seen = new AtomicReference<>();
CountDownLatch latch = new CountDownLatch(1);
logger.onLogEvent(CdpEventTypes.domMutation(mutation -> {
seen.set(mutation);
latch.countDown();
}));
WebElement img = driver.findElement(By.tagName("img"));
String newSrc = "img/award.png";
String script = String.format("arguments[0].src = '%s';", newSrc);
js.executeScript(script, img);
assertThat(latch.await(10, TimeUnit.SECONDS)).isTrue();
assertThat(seen.get().getElement().getAttribute("src"))
.endsWith(newSrc);
}
```

# Grid

[Dokumentacja](https://www.selenium.dev/documentation/grid/)

`Selenium Grid` to podprojekt pakietu Selenium, który pozwala na tworzenie sieciowej infrastruktury dla zdalnych przeglądarek dostępnych za pomocą protokołu W3C WebDriver. Selenium Grid umożliwia uruchamianie testów równolegle na różnych maszynach i różnych przeglądarkach. W tym celu Selenium Grid dostarcza serwer Selenium, którym można sterować za pomocą instancji RemoteWebDriver.

**Istnieją trzy sposoby uruchamiania Selenium Grid:**

1. **Standalone**
    - Jeden host działa jako serwer Selenium i dostarcza przeglądarki w tym trybie.
    - Zapewnia prosty sposób uruchamiania testów Selenium WebDriver w zdalnych przeglądarkach.
2. **Hub-nodes**
    - Problemem trybu standalone jest skalowalność, ponieważ serwer Selenium i przeglądarki działają na tym samym hoście.
    - Architektura hub-nodes definiuje dwa typy komponentów: hub (serwer Selenium) oraz jeden lub więcej węzłów (nodes) zarejestrowanych w hubie, które dostarczają przeglądarki do sterowania przez Selenium WebDriver.
3. **Tryb w pełni rozproszony**
    - Tryby standalone i hub-nodes to architektury scentralizowane, które mogą obniżać wydajność przy dużej liczbie żądań.
    - Od Selenium 4 dostępny jest tryb w pełni rozproszony, który implementuje mechanizmy równoważenia obciążenia, aby rozwiązać ten problem.

**Szczegóły dotyczące tych trybów i ich konfiguracji:**

**Standalone**

- Najprostszy sposób na uruchomienie infrastruktury Selenium Grid.
- Można uruchomić z poziomu shella lub kodu Java.

**Uruchamianie z powłoki:**

- Używa się binarnego pliku Selenium Grid w formie fat-JAR (uber-JAR), który zawiera wszystkie zależności.
- Plik ten można pobrać ze strony Selenium.
- Serwer Selenium automatycznie wykrywa dostępne sterowniki (np. chromedriver, geckodriver) w trybie standalone, szukając ich w zmiennej środowiskowej PATH.
- Zaleca się używanie WebDriverManager do automatycznego zarządzania sterownikami.
- WebDriverManager CLI jest dostępny jako fat-JAR do pobrania na GitHub.

**Przykład poleceń shell do pobrania chromedriver i geckodriver na Linuxie za pomocą WebDriverManager CLI oraz uruchomienia standalone grid:**

- Polecenia wykonuje się w tym samym folderze, aby sterowniki były dostępne dla Selenium Grid.

Po uruchomieniu standalone Selenium Server nasłuchuje na porcie 4444 localhosta.

Można utworzyć instancję RemoteWebDriver, używając URL [http://localhost:4444/](http://localhost:4444/) i odpowiednich capabilities (np. dla Chrome lub Firefox).

Dodatkowo Selenium Grid oferuje konsolę webową dostępną pod adresem serwera, która pozwala monitorować dostępne przeglądarki i aktywne sesje.

Przykład: Standalone Selenium Server może obsłużyć do ośmiu równoczesnych sesji (tyle ile jest dostępnych procesorów) dla Chrome i Firefox.

**Uruchamianie Selenium Grid z poziomu kodu Java**

Alternatywnym sposobem uruchomienia Selenium Grid jest użycie Javy. Oprócz fat-JAR, Selenium Grid jest dostępny w Maven Central z groupId: org.seleniumhq.selenium oraz artifactId: selenium-grid. W tym celu należy dodać odpowiednie zależności w konfiguracji projektu (Maven lub Gradle).

```xml
</dependency>

<dependency>
    <groupId>org.seleniumhq.selenium</groupId>
```

### Hub-nodes

Klasyczna architektura Selenium Grid obejmuje dwa typy hostów: hub (tj. Selenium Server) i grupę węzłów. Podobnie jak w trybie autonomicznym, możemy użyć Selenium Grid fat-JAR, aby uruchomić ten tryb w powłoce. Najpierw uruchamiamy hub na hoście. # Docker

Więcej na temat pracy z Dockerem na stronie 📄 [Docker](Docker%202e83f0d8989e80b2911dd454177d17b1.md) 

Docker to technologia oprogramowania typu open source, która umożliwia użytkownikom pakowanie i uruchamianie aplikacji jako lekkich, przenośnych kontenerów. Platforma Docker składa się z dwóch głównych komponentów: `Docker Engine`  - narzędzie do tworzenia i uruchamiania kontenerów, oraz `Docker Hub`  - usługa w chmurze do dystrybucji obrazów Docker. W domenie Selenium możemy używać Dockera do pakowania i uruchamiania skonteneryzowanych przeglądarek.

**WebDriver Manager i uruchamianie kontenerów Docker**

Począwszy od wersji 5, WebDriverManager umożliwia łatwe korzystanie z przeglądarek internetowych w kontenerach Docker. W tym celu każdy menedżer (np. chromedriver, firefoxdriver itp.) udostępnia metodę `browserInDocker()`. WebDriverManager wewnętrznie pobiera obrazy Docker i uruchamia kontener, tworząc instancję RemoteWebDriver podczas wywoływania metody `create()`. WebDriverManager korzysta z obrazów Docker utrzymywanych przez zespół Selenoid. W ten sposób można używać Chrome (stacjonarnego i mobilnego), Firefox, Edge, Opera i Safari jako kontenerów Docker po wyjęciu z pudełka za pośrednictwem WebDriverManager.

```java
class DockerChromeJupiterTest {
		WebDriver driver;
		WebDriverManager wdm = WebDriverManager.chromedriver().browserInDocker();
	@BeforeEach
		void setupTest() {
		assumeThat(isDockerAvailable()).isTrue();
		driver = wdm.create();
		}
		@AfterEach
	void teardown() {
		wdm.quit();
	}
	@Test
		void testDockerChrome() {
		driver.get("https://bonigarcia.dev/selenium-webdriver-java/");
		assertThat(driver.getTitle()).contains("Selenium WebDriver");
	}
}
```

💡 Po zakończeniu testu kontener zostaje wyłączony, jednak obraz jest cały czas obecny. 

WebDriverManager zapewnia płynny interfejs API do konfigurowania różnych aspektów dokowanych przeglądarek internetowych. Poniższy fragment pokazuje kilka możliwości. Więcej przykładów w repozytorium [Selenium na GitHubie](https://github.com/bonigarcia/selenium-webdriver-java). 

```java

```
