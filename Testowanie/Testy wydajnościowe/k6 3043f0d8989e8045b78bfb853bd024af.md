# k6 – Kompletne notatki

> Narzędzie do testów wydajnościowych i load testów. Skrypty piszesz w JS/TS, runtime k6 to binarny program napisany w Go.

## 1. Instalacja i setup z TypeScriptem

### Instalacja k6

```bash
# macOS
brew install k6

# Windows
winget install k6

# Linux (Debian/Ubuntu)
sudo gpg --no-default-keyring \
  --keyring /usr/share/keyrings/k6-archive-keyring.gpg \
  --keyserver hkp://keyserver.ubuntu.com:80 \
  --recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D69

echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main" \
  | sudo tee /etc/apt/sources.list.d/k6.list

sudo apt-get update && sudo apt-get install k6
```

### Setup projektu z TypeScriptem

k6 nie uruchamia TS natywnie – potrzebujesz bundlera (esbuild).

```bash
npm init -y
npm install -D typescript esbuild @types/k6
```

**tsconfig.json:**

```json
{
  "compilerOptions": {
    "target": "ES2015",
    "moduleResolution": "node",
    "lib": ["ES2015"],
    "strict": true
  }
}
```

**package.json – dodaj skrypty:**

```json
{
  "scripts": {
    "build": "esbuild src/test.ts --bundle --outfile=dist/test.js --platform=browser --target=es6",
    "test": "npm run build && k6 run dist/test.js"
  }
}
```

> Alternatywnie możesz sklonować oficjalny starter: `git clone https://github.com/grafana/k6-template-typescript`

## 2. Pierwszy test

```tsx
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Options } from 'k6/options';

export const options: Options = {
  vus: 10,         // wirtualni użytkownicy
  duration: '30s', // czas trwania
};

export default function () {
  const res = http.get('https://test.k6.io');

  check(res, {
    'status 200':     (r) => r.status === 200,
    'czas < 500ms':   (r) => r.timings.duration < 500,
  });

  sleep(1); // przerwa między iteracjami
}
```

```bash
npm run test
```

### Parametry wykonania

| Parametr | Opis |
| --- | --- |
| `execution: local` | Tryb uruchomienia: lokalnie lub w k6 Cloud |
| `script: test.js` | Plik skryptu |
| `output: -` | Domyślnie wyniki trafiają na stdout |
| `scenarios` | Konfiguracja scenariuszy (VU, czas, executor) |
| `gracefulStop: 30s` | Czas na dokończenie trwających iteracji na końcu testu |

### Wbudowane metryki k6

| Metryka | Opis |
| --- | --- |
| `http_req_duration` | **Całkowity czas odpowiedzi** (= sending + waiting + receiving). Najważniejsza metryka. |
| `{ expected_response:true }` | Czas odpowiedzi tylko dla udanych requestów (200-399). Używaj tej zamiast powyższej. |
| `http_req_waiting` | TTFB – czas do pierwszego bajtu od serwera |
| `http_req_sending` | Czas wysyłania danych do serwera |
| `http_req_receiving` | Czas odbierania odpowiedzi |
| `http_req_blocked` | Czas oczekiwania na wolne połączenie TCP |
| `http_req_connecting` | Czas nawiązania połączenia TCP |
| `http_req_tls_handshaking` | Czas TLS handshake |
| `http_req_failed` | % requestów zakończonych błędem (4xx/5xx) |
| `http_reqs` | Łączna liczba requestów + req/s |
| `iteration_duration` | Czas jednej pełnej pętli default() – uwzględnia sleep() |
| `iterations` | Łączna liczba iteracji + iter/s |
| `vus` / `vus_max` | Aktywni / maksymalni VU |
| `data_received` / `data_sent` | Transfer sieciowy |

### Jak czytać percentyle

Każda metryka raportowana jest jako: `avg`, `min`, `med`, `max`, `p(90)`, `p(95)`.

- **p(95) = 130ms** oznacza: 95% requestów miało czas odpowiedzi ≤ 130ms
- Używaj **p(95)** jako domyślnego punktu odniesienia – jest odporny na skrajne wartości
- Średnia (`avg`) może być myląca gdy są wartości odstające
- Przy jednym requeście wszystkie wartości będą identyczne

### Uwaga: http_req_failed – mylący output

```
http_req_failed: 0.00%  ✓ 0  ✗ 1
```

- ✓ = liczba requestów z `http_req_failed = true` (czyli prawdziwych błędów) → 0 błędów
- ✗ = liczba requestów z `http_req_failed = false` (czyli sukcesów) → 1 sukces

Wygląda odwrotnie, ale `✓ 0` oznacza brak błędów – to dobry wynik.

## 4. Checks (asercje)

`check()` weryfikuje odpowiedzi wewnątrz testu. **Nieudane checks NIE zatrzymują testu** – tylko liczą failures i raportują je na końcu. Wymagają świadomego połączenia z thresholdami żeby wpływały na wynik testu.

```tsx
import { check } from 'k6';
import http from 'k6/http';

export default function () {
  const res = http.get('https://api.example.com/users/1');

  check(res, {
    'status jest 200':      (r) => r.status === 200,
    'body zawiera id':      (r) => r.json('id') !== undefined,
    'czas odpowiedzi < 1s': (r) => r.timings.duration < 1000,
    'content-type jest JSON': (r) =>
      r.headers['Content-Type'].includes('application/json'),
  });
}
```

Wynik w raporcie:

```
checks.........................: 75.00% ✓ 3  ✗ 1
```

Jeśli chcesz żeby nieudane checks powodowały fail testu, połącz z thresholdem:

```tsx
thresholds: {
  checks: ['rate>=0.95'], // min 95% checks musi przejść
},
```

> **Dobra praktyka:** `check()` = weryfikacja poprawności odpowiedzi. `threshold` = definicja pass/fail testu. Używaj obu razem.

## 5. Opcje testów – vus, iterations, duration, stages

Opcje dodajesz po importach, przed funkcją `default`:

```tsx
export const options: Options = {
  vus: 10,
  duration: '2m',
};
```

### Iteracje

```tsx
export const options: Options = {
  vus: 10,
  iterations: 40, // łącznie 40 iteracji = 4 na VU
};
```

Opcja `iterations` definiuje łączną liczbę dla wszystkich VU. Przy `vus: 10` i `iterations: 40` każdy VU wykona skrypt 4 razy.

### Czas trwania

```tsx
export const options: Options = {
  vus: 10,
  duration: '2m', // VU-sy wykonują skrypt w pętli przez 2 minuty
};
```

Jeśli podasz obie opcje (`duration` i `iterations`), test kończy się gdy **pierwsza** zostanie osiągnięta.

Formaty czasu: `'30s'`, `'2m'`, `'1h30m'`, `'5m30s'`

### Stages (etapy) – kształtowanie obciążenia

Stages to najbardziej elastyczny sposób definiowania profilu obciążenia dla jednego scenariusza:

```tsx
export const options: Options = {
  stages: [
    { duration: '30m', target: 100 }, // 1. Ramp-up: 0 → 100 VU
    { duration: '1h',  target: 100 }, // 2. Steady state: 100 VU przez godzinę
    { duration: '5m',  target: 0   }, // 3. Ramp-down: 100 → 0 VU
  ],
};
```

Stages odpowiadają executorowi `ramping-vus` pod spodem.

## 6. Thresholds (progi)

Progi definiują **czy test przeszedł czy nie**. Gdy threshold jest niespełniony, k6 zwraca niezerowy kod wyjścia – co blokuje pipeline w CI/CD.

```tsx
export const options: Options = {
  thresholds: {
    http_req_failed:   ['rate<=0.05'],       // max 5% błędów
    http_req_duration: ['p(95)<=5000'],      // 95% requestów < 5s
    checks:            ['rate>=0.9'],        // min 90% checks musi przejść
  },
};
```

### Wskaźnik błędów

```tsx
thresholds: {
  http_req_failed: ['rate<=0.05'],
},
```

Orientacyjne wartości:

- Aplikacje krytyczne: ≤ 1%
- Aplikacje pomocnicze: ≤ 5%
- Testy chaos/awarii: 10-15%

### Czas odpowiedzi

Możesz używać: `avg`, `min`, `med`, `max`, `p(N)`. Zalecany punkt startowy: `p(95)`.

```tsx
thresholds: {
  // Pojedynczy próg
  http_req_duration: ['p(95)<=5000'],

  // Wiele progów dla tej samej metryki
  http_req_duration: ['p(90)<400', 'p(95)<800', 'p(99.9)<2000'],
},
```

### Progi na własnych metrykach

```tsx
thresholds: {
  'login_duration':   ['p(95)<1000'],
  'login_error_rate': ['rate<0.01'],
},
```

### Przerwanie testu przy niepowodzeniu progu

```tsx
thresholds: {
  http_req_failed: [{
    threshold:   'rate<=0.05',
    abortOnFail: true,         // zatrzymaj test natychmiast
    delayAbortEval: '10s',     // opcjonalnie: odczekaj 10s zanim przerwiesz
  }],
},
```

Przydatne gdy masz bardzo wysoki error rate spowodowany awarią komponentu – nie ma sensu dalej generować obciążenia.

## 7. Sleep – przerwy w testach

`sleep()` symuluje opóźnienia rzeczywistego użytkownika (czytanie strony, wypełnianie formularza).

```tsx
import { sleep } from 'k6';

export default function () {
  http.get('https://example.com');
  sleep(1);                          // stała przerwa 1s
  sleep(Math.random() * 5);          // losowo 0-5s
  sleep(randomIntBetween(1, 5));     // losowo liczba całkowita 1-5s
}
```

`randomIntBetween` pochodzi z `k6/x/faker` lub możesz napisać własną:

```tsx
function randomIntBetween(min: number, max: number): number {
  return Math.floor(Math.random() * (max - min + 1)) + min;
}
```

### Kiedy używać sleep

**Użyj sleep gdy:**

- Test symuluje sekwencję akcji użytkownika (flow)
- Chcesz odwzorować realistyczne zachowanie (czytanie, wypełnianie formularzy)
- CPU generatora obciążenia przekracza 80%

**Pomiń sleep gdy:**

- Szukasz maksymalnego throughput (stress test, szukanie limitu)
- Testujesz API bombardowane requestami bez opóźnień (np. job queue)
- CPU jest wystarczająco niskie bez sleep

> W razie wątpliwości – używaj sleep. Bez niego ryzykujesz fałszywe wyniki przez przeciążenie generatora obciążenia.

## 8. Zmienne środowiskowe

Przekazujesz przez flagę `-e` przy uruchamianiu. W skrypcie dostępne przez `__ENV`.

```bash
k6 run -e API_URL=https://api.example.com -e TOKEN=abc123 dist/test.js
```

```tsx
import http from 'k6/http';

export default function () {
  const res = http.get(`${__ENV.API_URL}/users`, {
    headers: { Authorization: `Bearer ${__ENV.TOKEN}` },
  });
}
```

> Nigdy nie hardcode'uj tokenów ani URL-i w skrypcie. Szczególnie ważne w CI/CD gdzie dane środowiskowe są zarządzane osobno.

## 9. Lifecycle hooks – setup i teardown

Specjalne funkcje wykonywane **raz** dla całego testu, poza VU-sami. Nie generują obciążenia.

```tsx
import http from 'k6/http';

// Wykonuje się RAZ przed testem (jeden VU, nie wszystkie)
export function setup() {
  const res = http.post('https://api.example.com/login', {
    username: 'testuser',
    password: 'secret',
  });

  const token = res.json('access_token') as string;
  return { token }; // zwrócone dane są dostępne w default() i teardown()
}

// Każdy VU dostaje dane z setup() jako argument
export default function (data: { token: string }) {
  http.get('https://api.example.com/me', {
    headers: { Authorization: `Bearer ${data.token}` },
  });
}

// Wykonuje się RAZ po teście – sprzątanie
export function teardown(data: { token: string }) {
  http.post('https://api.example.com/logout', null, {
    headers: { Authorization: `Bearer ${data.token}` },
  });
}
```

**Ważne:**

- `setup()` działa jako **jeden VU** – jego czas nie wlicza się do metryk testu
- Dane zwrócone z `setup()` muszą być serializowalne do JSON
- `teardown()` jest wykonywane nawet gdy test zostanie przerwany przez `abortOnFail`

## 10. Grupowanie requestów – group()

Pozwala logicznie grupować akcje użytkownika. Metryki są raportowane osobno per group, co ułatwia identyfikację wolnych kroków.

```tsx
import { group } from 'k6';
import http from 'k6/http';

export default function () {
  group('Logowanie', () => {
    http.post('/login', { user: 'test', pass: 'test' });
  });

  group('Przeglądanie katalogu', () => {
    http.get('/products');
    http.get('/products/123');
    http.get('/products/123/reviews');
  });

  group('Checkout', () => {
    http.post('/cart', JSON.stringify({ productId: 123 }));
    http.post('/order');
  });
}
```

W raporcie zobaczysz osobne metryki per group:

```
group_duration{group:::Logowanie}............: avg=120ms
group_duration{group:::Przeglądanie katalogu}: avg=280ms
group_duration{group:::Checkout}..............: avg=340ms
```

## 11. Własne metryki

k6 udostępnia 4 typy metryk, które możesz definiować i używać obok wbudowanych:

| Typ | Do czego | Przykład |
| --- | --- | --- |
| `Trend` | Rozkład wartości (avg, p95 itp.) | Czas operacji DB |
| `Counter` | Licznik rosnący | Liczba zalogowanych userów |
| `Rate` | Procent (ile razy warunek był true) | % błędnych odpowiedzi |
| `Gauge` | Bieżąca wartość (może rosnąć i maleć) | Aktualny rozmiar kolejki |

```tsx
import { Trend, Counter, Rate, Gauge } from 'k6/metrics';
import http from 'k6/http';

const loginDuration = new Trend('login_duration', true); // true = jednostka ms
const loginErrors   = new Rate('login_error_rate');
const loginTotal    = new Counter('login_total');

export default function () {
  const start = Date.now();
  const res   = http.post('/login', { user: 'test', pass: 'secret' });

  loginDuration.add(Date.now() - start);
  loginErrors.add(res.status !== 200);
  loginTotal.add(1);
}
```

Własne metryki możesz używać w thresholdach:

```tsx
thresholds: {
  'login_duration':   ['p(95)<1000'],
  'login_error_rate': ['rate<0.01'],
  'login_total':      ['count>100'], // min 100 loginów w teście
},
```

## 12. Parametryzacja danych – SharedArray

Gdy testujesz z wieloma różnymi userami/danymi, `SharedArray` wczytuje dane raz i współdzieli między wszystkimi VU-sami (zamiast każdy VU wczytywał osobno – oszczędność pamięci).

```tsx
import { SharedArray } from 'k6/data';
import http from 'k6/http';

// Wykonuje się JEDEN raz przy inicjalizacji, nie per VU
const users = new SharedArray('users', function () {
  return JSON.parse(open('./data/users.json'));
});

export default function () {
  // Każda iteracja losuje innego użytkownika
  const user = users[Math.floor(Math.random() * users.length)];

  const res = http.post('/login', {
    username: user.email,
    password: user.password,
  });
}
```

`data/users.json`:

```json
[
  { "email": "user1@test.com", "password": "pass1" },
  { "email": "user2@test.com", "password": "pass2" },
  { "email": "user3@test.com", "password": "pass3" }
]
```

`open()` wczytuje pliki relative do skryptu. Działa z JSON i CSV (CSV trzeba sparsować ręcznie lub przez bibliotekę).

## 13. Korelacja – wyciąganie danych z odpowiedzi

Częsty pattern w realistycznych testach: wyciągasz token/ID z jednego requestu i używasz go w kolejnych.

```tsx
export default function () {
  // 1. Login – wyciągnij token z odpowiedzi
  const loginRes = http.post('/auth/login', JSON.stringify({
    email:    'test@example.com',
    password: 'secret',
  }), {
    headers: { 'Content-Type': 'application/json' },
  });

  const token  = loginRes.json('data.access_token') as string;
  const userId = loginRes.json('data.user.id') as number;

  // 2. Użyj tokenu w kolejnych requestach
  const headers = { Authorization: `Bearer ${token}` };

  const profileRes = http.get(`/users/${userId}`, { headers });
  check(profileRes, { 'profil załadowany': (r) => r.status === 200 });

  // 3. Wyciągnij ID z kolejnej odpowiedzi i użyj dalej
  const ordersRes = http.get('/orders', { headers });
  const firstOrderId = ordersRes.json('orders[0].id') as number;

  http.get(`/orders/${firstOrderId}`, { headers });
}
```

## 14. Zaawansowane scenariusze – executory

Stages to uproszczona wersja executora `ramping-vus`. Pełna lista executorów – każdy pasuje do innego przypadku:

| Executor | Kiedy używać |
| --- | --- |
| `constant-vus` | Stała liczba userów przez czas – klasyczny load test |
| `ramping-vus` | Zmienna liczba userów – to co robią `stages` |
| `constant-arrival-rate` | Stała liczba req/s niezależnie od VU – testujesz throughput |
| `ramping-arrival-rate` | Rosnący RPS – szukasz punktu krytycznego aplikacji |
| `shared-iterations` | Pula iteracji do wyczerpania przez wszystkich VU |
| `per-vu-iterations` | Każdy VU wykonuje dokładnie X iteracji |

### Kluczowa różnica: VU-based vs arrival-rate

- `constant-vus` – masz 10 userów, każdy robi ile zdąży → throughput zależy od szybkości serwera
- `constant-arrival-rate` – **dokładnie 100 req/s** niezależnie od serwera → VU-sy dobierane automatycznie

```tsx
export const options: Options = {
  scenarios: {
    api_load: {
      executor:        'constant-arrival-rate',
      rate:            100,
      timeUnit:        '1s',    // 100 req/s
      duration:        '5m',
      preAllocatedVUs: 50,
      maxVUs:          200,     // k6 doda więcej VU jeśli potrzeba
    },
  },
};
```

### Wiele scenariuszy jednocześnie

```tsx
export const options: Options = {
  scenarios: {
    zwykli_uzytkownicy: {
      executor: 'ramping-vus',
      startVUs: 0,
      stages: [
        { duration: '2m', target: 50 },
        { duration: '5m', target: 50 },
        { duration: '1m', target: 0  },
      ],
      exec: 'browseApp', // wskazuje funkcję do uruchomienia
    },
    api_boty: {
      executor: 'constant-arrival-rate',
      rate:     200,
      timeUnit: '1s',
      duration: '5m',
      preAllocatedVUs: 30,
      exec: 'callApi',
    },
  },
};

export function browseApp() {
  http.get('/products');
  sleep(randomIntBetween(1, 3));
}

export function callApi() {
  http.get('/api/v1/status');
}
```

## 15. Organizacja kodu – większe projekty

### Struktura katalogów

```
src/
  api/
    users.ts       # funkcje dla endpointów users
    orders.ts      # funkcje dla endpointów orders
  flows/
    checkout.ts    # kompletny user flow
  data/
    users.json     # dane testowe
  utils/
    auth.ts        # helper do logowania
  scenarios/
    smoke.ts       # test dymny
    load.ts        # test obciążeniowy
    stress.ts      # test przeciążeniowy
```

### Przykład – wydzielenie endpointów

```tsx
// src/api/users.ts
import http from 'k6/http';

const BASE_URL = __ENV.API_URL || 'https://api.example.com';

export function getUser(id: number, token: string) {
  return http.get(`${BASE_URL}/users/${id}`, {
    headers: { Authorization: `Bearer ${token}` },
  });
}

export function createUser(payload: object, token: string) {
  return http.post(`${BASE_URL}/users`, JSON.stringify(payload), {
    headers: {
      'Content-Type':  'application/json',
      'Authorization': `Bearer ${token}`,
    },
  });
}
```

```tsx
// src/scenarios/load.ts
import { getUser, createUser } from '../api/users';

export const options = { /* ... */ };

export function setup() { /* login, zwróć token */ }

export default function (data: { token: string }) {
  getUser(1, data.token);
  sleep(1);
}
```

## 16. Zapisywanie i eksport wyników

## Dobre praktyki – podsumowanie

- Zawsze definiuj **thresholds** – bez nich test nigdy nie "nie przejdzie" automatycznie

## 19. Tags – etykietowanie requestów

W przeciwieństwie do `group()`, tagi **nie tworzą zagregowanych metryk** – służą do dodawania metadanych, które możesz filtrować w wynikach (CSV, Grafana itp.).

### Tagowanie pojedynczego requestu

```tsx
import http from 'k6/http';

export default function () {
  http.get('http://ecommerce.k6.io/', {
    tags: {
      page: 'Homepage',
      type: 'HTML',
      env:  'staging',
    },
  });
}
```

W wynikach CSV pojawi się kolumna `extra_tags`:

```
page=Homepage&type=HTML&env=staging
```

### URL grouping – grupowanie dynamicznych URL-i

Gdy skrypt odwiedza wiele podobnych URL-i (np. różne strony produktów), domyślnie każdy URL ma osobną metrykę. Żeby połączyć je w jedną, użyj tagu `name`:

```tsx
const products = ['album', 'beanie', 'hoodie'];
const rand = Math.floor(Math.random() * products.length);

http.get(`http://example.com/product/${products[rand]}`, {
  tags: { name: 'ProductPage' }, // wszystkie warianty raportowane jako jeden
});
```

Bez URL grouping: metryki per `/product/album`, `/product/beanie`, itd.
Z URL grouping: jedna metryka `ProductPage` dla wszystkich.

### Co można tagować

- requesty HTTP
- checks
- thresholds
- własne metryki
- wszystkie metryki w teście (przez opcje)

> `group()` = agreguje metryki i raportuje `group_duration`. `tags` = dodaje metadane, bez agregacji. Używaj obu razem.
>

## 20. Parallel requests – równoległe requesty

Domyślnie k6 wysyła requesty sekwencyjnie. Przeglądarki robią to równolegle – jeśli chcesz to symulować, użyj `http.batch()`.

```tsx
import http from 'k6/http';
import { check } from 'k6';

const domain = 'https://test.k6.io';

export default function () {
  // Wszystkie requesty wysłane równocześnie
  const responses = http.batch([
    ['GET', `${domain}/`],
    ['GET', `${domain}/static/css/site.css`],
    ['GET', `${domain}/static/js/prisms.js`],
    ['GET', `${domain}/static/favicon.ico`],
  ]);

  check(responses[0], {
    'Strona główna załadowana': (r) =>
      r.body.includes('Collection of simple web-pages'),
  });
}
```

### Kiedy używać batch

- Testujesz stronę www i chcesz uwzględnić pobieranie zasobów statycznych (CSS, JS, obrazki)
- Wiesz że w produkcji przeglądarka pobiera kilka zasobów równocześnie

### Kiedy NIE używać batch

- Testujesz API które wywołuje requesty sekwencyjnie
- Nie masz pewności jak działa produkcja – sprawdź DevTools w przeglądarce

> Równoległe requesty zwiększają RPS i mogą podnieść wykorzystanie CPU zarówno na generatorze obciążenia jak i serwerze aplikacji.
>

## 21. Execution context variables

Moduł `k6/execution` udostępnia zmienne opisujące aktualny stan testu. Przydatne gdy chcesz przypisać unikalne dane do konkretnego VU lub debugować scenariusze.

### Główne zmienne

```tsx
import { vu, scenario, instance, test } from 'k6/execution';

export default function () {
  console.log('VU ID w teście:', vu.idInTest);       // unikalny numer VU (globalnie)
  console.log('Numer iteracji:',  vu.iterationInScenario);
  console.log('Nazwa scenariusza:', scenario.name);
  console.log('Aktywny executor:', scenario.executor);
}
```

Dostępne też stare globalne zmienne (bez importu):

- `__VU` – numer VU (przestarzałe, ale działa)
- `__ITER` – numer iteracji danego VU

### Praktyczny przykład: unikalny user per VU

Problem: aplikacja blokuje wielokrotne logowanie tego samego usera przez różne VU.
Rozwiązanie: każdy VU dostaje własne konto z pliku CSV.

```tsx
import papaparse from 'https://jslib.k6.io/papaparse/5.1.1/index.js';
import { SharedArray } from 'k6/data';
import { vu } from 'k6/execution';

const users = new SharedArray('users', function () {
  return papaparse.parse(open('./data/users.csv'), { header: true }).data;
});

export const options = {
  scenarios: {
    login: {
      executor:   'per-vu-iterations',
      vus:        users.length, // jeden VU na usera
      iterations: 1,
    },
  },
};

export default function () {
  // vu.idInTest zaczyna od 1, tablica od 0 → stąd -1
  const user = users[vu.idInTest - 1];
  console.log(`VU ${vu.idInTest} loguje się jako: ${user.username}`);
  // http.post('/login', { username: user.username, password: user.password });
}
```

### Inne zastosowania

- Warunkowa logika per scenariusz: `if (scenario.name === 'stress_test') { ... }`
- Przerywanie testu z komunikatem: `test.abort('Za dużo błędów!')`
- Dodawanie tagów specyficznych dla VU

## 23. Debugowanie – zaawansowane

### Strategia debugowania krok po kroku

1. **Dodaj checks** do każdej ważnej akcji – bez nich k6 nie poinformuje o błędzie logiki aplikacji, tylko o błędach HTTP
2. **Dodaj console.log()** dla kluczowych zmiennych
3. **Użyj --http-debug** gdy sprawdzasz co dokładnie jest wysyłane
4. **Uruchom z 1 VU, 1 iteracją** żeby izolować problem

```bash
k6 run --vus 1 --iterations 1 dist/test.js
k6 run --vus 1 --iterations 1 --http-debug dist/test.js       # headers
k6 run --vus 1 --iterations 1 --http-debug=full dist/test.js  # headers + body
```

### Pominięcie setup/teardown podczas debugowania

```bash
k6 run --no-setup --no-teardown dist/test.js
```

Przydatne gdy setup jest wolny (np. tworzy dane testowe) a chcesz szybko sprawdzić logikę VU.

> **Uwaga:** Wyłącz `console.log()` przed uruchomieniem właściwego testu. Nadmierne logowanie jest resource-intensive i może zafałszować wyniki.
> 

### Pułapki debugowania

| Problem | Jak wykryć |
| --- | --- |
| Brak CSRF tokenu | `--http-debug=full` → `error: invalid csrf token` w body |
| Redirect 308 zamiast 200 | `--http-debug` → widać dodatkowy request |
| Błędne dane logowania | `console.log()` przy zmiennych z losowaniem |
| Check przechodzi ale aplikacja zwraca błąd | Sprawdź czy check weryfikuje właściwą rzecz |

## 24. Recording – nagrywanie skryptu

Zamiast pisać skrypt ręcznie, możesz nagrać sesję w przeglądarce. k6 oferuje **darmowe** narzędzie do tego.

### Kiedy nagrywać vs pisać ręcznie

**Nagrywaj gdy:**

- Testujesz stronę www z wieloma zasobami statycznymi (CSS, JS, obrazki)
- Nie znasz dobrze struktury HTTP requestów aplikacji
- Aplikacja ma dużo dynamicznych wartości do korelacji

**Pisz ręcznie gdy:**

- Testujesz API (znasz endpointy)
- Flow jest prosty lub dobrze udokumentowany
- Chcesz precyzyjną kontrolę nad każdym requestem

### Jak nagrać

1. Zainstaluj **k6 Browser Recorder** ([Chrome](https://chrome.google.com/webstore/detail/k6-browser-recorder/phjdhndljphphehjpgbmpocddnnmdbda) / Firefox)
2. Utwórz darmowe konto na **k6 Cloud** (nie wymaga karty)
3. Kliknij ikonę rozszerzenia → skonfiguruj opcje:
    - `Download HAR file` – zapisuje plik HAR lokalnie
    - `Clear cache` – symuluje nowego użytkownika
    - `Correlate request/response data` – automatyczna korelacja
4. Kliknij **Start recording** i przejdź przez flow w przeglądarce
5. Kliknij **Stop** → jesteś przekierowany do k6 Cloud
6. Skopiuj wygenerowany skrypt

### Ważne opcje przy zapisie

- `Include static assets` – dołącza requesty do CSS/JS/obrazków
- `Generate sleep` – dodaje sleep na podstawie rzeczywistych opóźnień podczas nagrania
- `Third-party domains filtering` – ostrożnie: nie testuj serwisów zewnętrznych których nie posiadasz

### Co zrobić po nagraniu

Wygenerowany skrypt to **punkt startowy**, nie gotowy test. Sprawdź:

- Dodaj/popraw **checks** do weryfikacji odpowiedzi
- Zmień **sleep** z hardcodowanych wartości na dynamiczne (`Math.random()`)
- Sprawdź czy jest potrzebna **korelacja** (tokeny CSRF, session ID itp.)
- Ustaw **opcje testowe** (VU, duration, thresholds)
- Przeorganizuj kod używając **group()** i funkcji

## 26. Workload modeling – modelowanie obciążenia

Workload modeling to przemyślenie **JAK** aplikować obciążenie – nie tylko co testować, ale jak realistycznie to symulować.

### Elementy modelu obciążenia

**Parametry testu** – liczba VU, czas, iteracje, etapy (omówione w sekcji 5 i 14)

**Think time i pacing** – opóźnienia między akcjami (sekcja 7). Dłuższe → mniej requestów. Używaj losowych wartości.

**Zasoby statyczne** – czy skrypt ma pobierać CSS/JS/obrazki razem z HTML? Tak = bardziej realistyczne dla web, Nie = testujesz tylko API/backend

**Parallel requests** – przeglądarki pobierają wiele zasobów jednocześnie (sekcja 20). Używaj `http.batch()` gdy to odwzorowuje produkcję.

**Cache i cookies** – k6 domyślnie nie cachuje. Możesz zarządzać tym przez headers i opcje cookies. Decyzja zależy od celu: nowy user (brak cache) vs powracający (cache włączony).

**Dane testowe** – używanie tych samych danych wielokrotnie może powodować caching po stronie serwera i fałszować wyniki. Używaj SharedArray z różnymi danymi (sekcja 12).

### Dlaczego realistyczny workload ma znaczenie

Nierealistyczny skrypt może:

- nie osiągnąć oczekiwanego throughput
- testować inne komponenty niż w produkcji
- dać fałszywe poczucie bezpieczeństwa

> Najczęstszy błąd: incydenty produkcyjne mimo "udanych" testów obciążeniowych – klasyczny objaw nierealistycznego workload modelingu.
>

## 27. Zaawansowane opcje scenariuszy

Kilka opcji które nie były omówione przy executorach:

```tsx
export const options = {
  scenarios: {
    browsing: {
      executor:     'ramping-vus',
      startVUs:     0,
      stages: [
        { duration: '2m', target: 50 },
        { duration: '5m', target: 50 },
        { duration: '1m', target: 0  },
      ],
      gracefulRampDown: '30s',  // czas na dokończenie iteracji podczas ramp-down
      exec:             'browse',
      startTime:        '0s',   // kiedy zacząć ten scenariusz
      env:  { ENV_TYPE: 'load' }, // zmienne środowiskowe specyficzne dla scenariusza
      tags: { scenario: 'browsing' }, // tagi dla wszystkich metryk w scenariuszu
    },
    api_calls: {
      executor:  'constant-arrival-rate',
      rate:      50,
      timeUnit:  '1s',
      duration:  '5m',
      preAllocatedVUs: 20,
      startTime: '30s', // zaczyna 30s po starcie testu – daje czas browsing na rozgrzewkę
      exec: 'callApi',
    },
  },
};
```

### startTime – opóźnienie startu scenariusza

Pozwala symulować scenariusze uruchamiane falami. Np. najpierw generuj ruch "przeglądania", a po 30 sekundach dodaj ruch "kupowania".

### gracefulRampDown

Podczas ramp-down (redukcji VU) k6 może przerwać trwające iteracje. `gracefulRampDown: '30s'` daje 30 sekund na dokończenie.

## 28. k6 Browser – testy UI i przeglądarki

Moduł browser dodaje automatyzację przeglądarki i testy end-to-end do k6, zachowując przy tym wszystkie podstawowe funkcje k6. API modułu jest wzorowane na Playwright – jeśli znasz Playwright, będziesz się tu czuł jak w domu.

Od wersji k6 0.52.0 moduł browser jest wbudowany w k6 jako moduł core – nie wymaga osobnej instalacji ani kompilacji.

### Kiedy używać browser vs protokół HTTP

Możesz mieszać testy na poziomie przeglądarki i protokołu HTTP w jednym skrypcie. Typowe podejście: generuj większość ruchu przez testy protokołu (HTTP), a jeden lub dwa VU uruchamiaj na poziomie przeglądarki żeby symulować prawdziwego użytkownika.

| Poziom | Kiedy |
| --- | --- |
| **HTTP (protokół)** | Testy API, duże obciążenie, max throughput |
| **Browser** | Weryfikacja UI pod obciążeniem, Core Web Vitals, flow wymagające przeglądarki (OAuth, captcha) |

### Wymagania

- Zainstalowany k6 (najnowsza wersja)
- Chromium lub Google Chrome na maszynie

### Pierwszy test browser

API od wersji 0.52.0 jest asynchroniczne – funkcja `default` musi być `async`.

```tsx
import { browser } from 'k6/browser';
import { check } from 'k6';

export const options = {
  scenarios: {
    ui: {
      executor: 'shared-iterations',
      options: {
        browser: {
          type: 'chromium', // jedyna obsługiwana opcja na dziś
        },
      },
    },
  },
  thresholds: {
    checks: ['rate==1.0'],
  },
};

export default async function () {
  const page = await browser.newPage();

  try {
    await page.goto('<https://test.k6.io/>');

    // Kliknięcie w element
    await page.locator('a[href="/contacts.php"]').click();

    // Weryfikacja
    const header = await page.locator('h2').textContent();
    check(page, {
      'nagłówek widoczny': () => header === 'Contact us',
    });

    // Screenshot (przydatny do debugowania)
    await page.screenshot({ path: 'screenshots/screenshot.png' });

  } finally {
    await page.close(); // ZAWSZE zamykaj page w finally – inaczej Web Vitals nie są poprawnie obliczane
  }
}
```

```bash
k6 run dist/test.js
```


| --- | --- |
| `browser_web_vital_lcp` | Largest Contentful Paint – czas ładowania głównej treści |
| `browser_web_vital_fid` | First Input Delay – czas reakcji na pierwszą interakcję |
| `browser_web_vital_cls` | Cumulative Layout Shift – stabilność layoutu |
| `browser_web_vital_fcp` | First Contentful Paint – czas do pierwszego wyrenderowania |
| `browser_web_vital_ttfb` | Time to First Byte – czas do pierwszego bajtu |
| `browser_web_vital_inp` | Interaction to Next Paint – responsywność strony |

Możesz ustawić na nie thresholdy:

```tsx
thresholds: {
  'browser_web_vital_lcp': ['p(75)<2500'], // Google zaleca < 2.5s
  'browser_web_vital_cls': ['p(75)<0.1'],  // Google zaleca < 0.1
},
```

### Symulacja urządzeń mobilnych

```tsx
import { browser, devices } from 'k6/browser';

export default async function () {
  const iphoneX  = devices['iPhone X'];
  const context  = await browser.newContext(iphoneX); // viewport, UA, itp.
  const page     = await context.newPage();

  try {
    await page.goto('<https://example.com>');
  } finally {
    await page.close();
  }
}
```

### Dobre praktyki

- **Zawsze używaj `try/finally` z `page.close()`** – bez zamknięcia page metryki Web Vitals są niepoprawne
- **Nie uruchamiaj setek VU w trybie browser** – każdy VU to osobna instancja Chromium, bardzo zasobochłonne. Browser VU = 1-5, resztę rób przez HTTP
- **Używaj `page.waitForNavigation()`** razem z kliknięciami które powodują nawigację – inaczej test może nie poczekać na załadowanie strony
- **Screenshoty przy failach** – dodaj screenshot w bloku `catch` żeby wiedzieć co się działo gdy test padł
- **Wyłącz logowanie** przy większych testach – browser generuje dużo logów

## Kluczowe pojęcia:

**Core Web Vitals** to zestaw metryk Google mierzących *jakość doświadczenia użytkownika* na stronie:

**LCP – Largest Contentful Paint** – czas do momentu gdy największy element na stronie (zwykle główny obrazek albo nagłówek) jest w pełni wyrenderowany. Mierzy *postrzegane tempo ładowania*. Google zaleca < 2.5s. Jeśli LCP rośnie pod obciążeniem, znaczy że serwer wolniej serwuje zasoby.

**CLS – Cumulative Layout Shift** – mierzy jak bardzo elementy strony "skaczą" podczas ładowania (np. obrazek się ładuje i przesuwa tekst który już czytasz). To liczba bezwymiarowa, nie czas. Google zaleca < 0.1. Pod obciążeniem może rosnąć gdy zasoby ładują się w innej kolejności.

**FCP – First Contentful Paint** – czas do pierwszego czegokolwiek wyrenderowanego na ekranie (cokolwiek – tekst, obrazek). Użytkownik przestaje widzieć białą stronę.

**TTFB – Time to First Byte** – czas od wysłania requestu do otrzymania pierwszego bajtu odpowiedzi. To metryka serwera, nie przeglądarki. Najwrażliwsza na obciążenie backendu.

**FID – First Input Delay** – czas od pierwszej interakcji użytkownika (klik, tap) do momentu gdy przeglądarka zaczyna na nią reagować. Mierzy responsywność.

**INP – Interaction to Next Paint** – nowszy następnik FID, mierzy czas reakcji przez całą sesję, nie tylko pierwsze kliknięcie.

## Darmowe API do ćwiczeń

Kilka sprawdzonych źródeł:

**Dedykowane do load testingu (oficjalne środowiska testowe):**

- **test.k6.io** – własny serwis k6, stworzony właśnie do testów ładowania

- **jsonplaceholder.typicode.com** – fałszywe REST API (posty, komentarze, usery), działa jak prawdziwy CRUD
- **reqres.in** – realistyczne API z loginami, rejestracją, paginacją

**Prawdziwe publiczne API:**

- **api.open-meteo.com** – pogoda, bez klucza API, stabilne
- **pokeapi.co** – Pokémon API, duża baza, dobre do testów GET z różnymi ID
- **dummyjson.com** – produkty, koszyk, logowanie, bardzo kompletne
- **fakestoreapi.com** – sklep, ma auth, produkty, koszyk

**Do testowania autoryzacji i bardziej złożonych flow:**

- **the-internet.herokuapp.com** – loginy, formularze, CSRF, zaprojektowane pod testy
- **practicesoftwaretesting.com** – pełny sklep z API, rejestracja, checkout

**Ważna uwaga:** przy publicznych API pamiętaj żeby nie przesadzać z obciążeniem – stawiaj niskie VU (1-5) i dodawaj `sleep`. Część z nich ma rate limiting. Do poważniejszych testów lepiej postawić własne API lokalnie np. przez **json-server** (`npm install -g json-server`) – sekundę konfiguracji i masz własne REST API z pliku JSON, bez żadnych ograniczeń.

**Oficjalne środowiska:**

- **k6 Demo Store** (`ecommerce.test.k6.io`) – sklep WooCommerce, ma login, koszyk, checkout, grupy requestów
- **Sauce Labs: The Internet** (`the-internet.herokuapp.com`) – kolekcja stron z różnymi wzorcami: formularze, auth, CSRF, dynamiczne elementy
- **OWASP WebGoat** – celowo dziurawa aplikacja, dobra do nauki korelacji i bardziej złożonych flow (instalujesz lokalnie przez Docker)

**Aplikacje do samodzielnego postawienia (Docker):**

- **httpbin** – `docker run -p 80:80 kennethreitz/httpbin` → masz własne echo API lokalnie, zero limitów
- **Mockoon** – desktopowa aplikacja do tworzenia mock API w minutę, bez kodowania
- **Prism (Stoplight)** – generuje mock API z pliku OpenAPI/Swagger, jeśli masz specyfikację swojego API to od razu masz środowisko testowe
- **WireMock** – bardziej rozbudowany mock server, popularny w środowiskach enterprise
